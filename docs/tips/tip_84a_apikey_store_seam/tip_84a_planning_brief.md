Decision basis: recon - the authenticator is already behind `ILocalDevApiKeyAuthenticator` (endpoints depend on it), but the KEY STORE is NOT behind an interface: `LocalDevApiKeyValidator` binds the CONCRETE `LocalDevRuntimePolicySource` and reads `.ApiKeys` (`LocalDevApiKeyValidator.cs:6,15`), and that class doubles as both the key store AND `ILocalDevClientPolicyProvider`. Category/scope authorization is already cleanly DECOUPLED (reads only `AuthenticatedClientContext`). So production-auth = a store-swap behind the authenticator/validator; this slice introduces that seam. Implements the TIP-15 boundary behind the TIP-13 `ApplicationAuthorization` seam (TIP-15 section-16 candidate).

## 1. Intent
Introduce a neutral `IApiKeyStore` seam so a future production credential store (Slice-2: hashed persistent store) can plug in WITHOUT touching authorization, and split the key-store responsibility out of `LocalDevRuntimePolicySource` so the client-policy provider is untouched. This slice is a PURE structural refactor: identical auth outcomes + identical public codes; NO hashing, NO production store, NO lifecycle (those are Slice-2/3, gated on resolving TIP-15's open STOP/RRI). It preserves every TIP-12/TIP-13 rule and honors every TIP-15 locked invariant.

## 2. Scope - ALLOWED

### 2.1 Introduce `IApiKeyStore` (neutral key-lookup seam)
- New interface (neutral namespace, NOT `LocalDev`) whose single lookup takes the PRESENTED key and returns the matched credential record (or null): `Task<ResolvedApiKey?> FindByPresentedKeyAsync(string presentedApiKey, CancellationToken ct)`.
- **[self-check] ASYNC (Task-returning), NOT sync** - even though the LocalDev impl is in-memory (returns `Task.FromResult`), the Slice-2/TIP-84B production store is DB-backed (I/O = async). Making the seam async NOW avoids an interface change in TIP-84B (the whole point of the seam).
- **[review P2] Async signatures pinned end-to-end WITH the existing `CancellationToken`** (endpoints already carry one, `VerificationSessionEndpoints.cs:29`): `IApiKeyStore.FindByPresentedKeyAsync(string presentedApiKey, CancellationToken ct)`; `LocalDevApiKeyValidator.ValidateAsync(string? apiKeyValue, string? requiredScope, CancellationToken ct)`; `IApiKeyAuthenticator.AuthenticateAsync(HttpContext httpContext, string? requiredScope, CancellationToken ct)`. ALL 9 endpoints `await authenticator.AuthenticateAsync(..., cancellationToken)` (flow the endpoint's CT, do NOT drop it - else TIP-84B's DB store loses request cancellation). This ripple is mechanical/structural, no behavior change.
- **[self-check] The returned `ResolvedApiKey` carries credential METADATA ONLY - NO plaintext key value.** The store MATCHES internally (LocalDev = plaintext `SingleOrDefault` `LocalDevApiKeyValidator.cs:15`; TIP-84B = prefix/id lookup + constant-time hash-verify), then returns `{apiKeyId, clientApplicationId, keyPrefix, scopes, status, expiresAt, callerCategory, allowed*Ids}` - WITHOUT `ApiKeyValue`. Slice maps `LocalDevApiKeyRecord` -> `ResolvedApiKey` dropping the plaintext.
- **[review P2] `ResolvedApiKey` is a NEW auth-resolution type - do NOT reuse the existing `ApiKeyMetadata`** (`src/TagEkyc.Domain/ApiKeyMetadata.cs`): it lacks `CallerCategory`/`AllowedClientApplicationIds`/`AllowedCaptureAgentIds` and reusing it would weaken the `AuthenticatedClientContext`.
- **[review P1-03] Preserve TIP-15 principal/credential separation:** `ResolvedApiKey` keeps `apiKeyId` (the rotating CREDENTIAL id) and `clientApplicationId` as DISTINCT fields - do NOT collapse them. This slice does NOT introduce `principalId` (that is TIP-84B/84C per TIP-15), but it MUST NOT bake `apiKeyId` as the permanent principal; keep the fields separable so TIP-84B/84C can add `principalId` without a rewrite.
- **[review P1-02/P3] `presentedApiKey` = SECRET INPUT ONLY:** it flows INTO the store-lookup/validation call and MUST NOT be logged, stored, or returned; `ResolvedApiKey` never contains it. The production (TIP-84B) impl SHOULD derive a lookup selector/prefix internally then constant-time hash-verify (not scan all rows). `null` return = no valid credential match. (This resolves the apparent tension with "no raw-secret-across-boundaries": the presented key crosses ONLY as verification input, never as output/state.)
- **Interface ANTICIPATES hash-lookup (TIP-84B) without an interface change:** it takes the presented key and lets the IMPL decide HOW to match; do NOT bake plaintext-equality into the interface contract.

### 2.2 Split the key store OUT of `LocalDevRuntimePolicySource`
- Extract the API-key store role into a `LocalDevApiKeyStore` (implements `IApiKeyStore`, holds the current seeded `LocalDevApiKeyRecord` list). `LocalDevRuntimePolicySource` KEEPS its `ILocalDevClientPolicyProvider` role (client policies / allowed scopes) UNTOUCHED - that interface is already consumed by app services and must not change.
- **[review P1 - CRITICAL] `LocalDevApiKeyValidator` depends on BOTH `IApiKeyStore` AND the existing `ILocalDevClientPolicyProvider`, NOT just the store.** The KEY credential lookup moves to `IApiKeyStore`; the **client-application disabled/status check STAYS via the policy provider** - the current `CLIENT_APPLICATION_DISABLED` check reads `policies.Policies` (`LocalDevApiKeyValidator.cs:31`), which is the POLICY source, not the key store. Do NOT move client-policy/status into `ResolvedApiKey`, and do NOT drop the disabled-client check. The validator's status/expiry/scope logic stays byte-for-byte; `CLIENT_APPLICATION_DISABLED` remains public-code-equivalent.
- DI: register `IApiKeyStore -> LocalDevApiKeyStore`; the LocalDev store stays a DEV/LocalDev backend (TIP-15: LocalDev keys never promoted to production).

### 2.3 Neutral request-path authenticator seam (rename off `LocalDev`)
- Rename the request-path SEAM interface `ILocalDevApiKeyAuthenticator` -> `IApiKeyAuthenticator` (neutral, OQ-3 RESOLVED = in this slice); the 9 endpoint injections + DI registration re-point to it. The CONCRETE `LocalDevApiKeyAuthenticator` keeps its name (it IS the localdev backend) and implements the neutral interface. MECHANICAL rename, no logic change.

### 2.4 NO behavior change (the core invariant of this slice)
- **Full no-behavior-change matrix (identical outcomes + public codes + messages):** missing/malformed API-key header; unknown/no-match key; `API_KEY_REVOKED` (401); `API_KEY_EXPIRED` (401); `CLIENT_APPLICATION_DISABLED` (401, via policy provider 2.2); `MISSING_SCOPE` (403); TIP-13 `CALLER_CATEGORY_NOT_ALLOWED`; tenant `FORBIDDEN_CLIENT_APPLICATION`. Each identical to today.
- **[review P1-05] Endpoint behavior identical (blast-radius guard):** authenticate still runs BEFORE any body-dependent command; the correlation-id / error-envelope shape is unchanged; NO scope/category check moves from the app service to the endpoint; NO extra `await` inserted after a mutation begins but before auth. The async ripple is signature-only.
- Same lookup semantics (plaintext equality behind the LocalDev impl), same status/expiry checks, same `AuthenticatedClientContext` output. A structural diff only.

## 3. STOP / NOT
- NO auth behavior/logic change - same outcomes, same public codes, same context; this slice is the SEAM only (verify by keeping all existing auth tests green + a no-behavior-change diff).
- NO production key store / NO hashing / NO constant-time-compare implementation / NO lifecycle (issuance/rotation/revocation) here - those are Slice-2/Slice-3 and are GATED on resolving TIP-15's open STOP/RRI (hashing scheme, secret-store posture, rotation authority, IdP). Do NOT pre-empt them.
- NO change to `ApplicationAuthorization`, the category+scope model, the TIP-12 ownership/access matrix, cross-client-forbidden, or `AuthenticatedClientContext` (TIP-13 stays intact).
- NO change to `ILocalDevClientPolicyProvider` / `ValidatePolicy` (client-policy provider stays; only the key store is extracted from the same class).
- NO promoting/migrating LocalDev seeded keys to production (TIP-15 invariant) - the LocalDev store stays a dev backend.
- NO re-deciding the TIP-15 boundary or its invariants; NO IdP/secret-store choice here.
- NO conflation with the evidence-SIGNING key custody (D-P0-1 HSM/PKCS#11, `IEvidenceSigner`) - that is a DIFFERENT key domain; this slice is auth API-credentials only.
- NO `TagEkyc.SignFlow`/consumer-specific reference in the Application assembly (TIP-13 ArchTest stays green).

## 5. Review tier & attacks
Tier-2 (structural refactor, but on the auth path). Attacks: (a) did any auth OUTCOME or public code change (must not)? (b) does `IApiKeyStore` genuinely enable a store swap (presented-key -> record, impl-decides-match), or does it leak plaintext-equality into the interface? (c) is `ILocalDevClientPolicyProvider` / `ValidatePolicy` truly untouched? (d) any TIP-12/13 rule (category/scope/ownership/cross-client) loosened? (e) any TIP-15 invariant violated (LocalDev key promoted, raw-secret rule pre-empted, principal/credential model contradicted)? (f) did the slice accidentally start implementing hashing/store/lifecycle (scope creep into TIP-84B/C)? (g) any conflation with the signing-key (HSM) domain? (h) **is `CLIENT_APPLICATION_DISABLED` still enforced via the policy provider (not lost when the store split)?** (i) is `presentedApiKey` ever logged/stored/returned (must not)? (j) is `apiKeyId` baked as the permanent principal (must stay separable from clientApplicationId)? (k) is the endpoint `CancellationToken` dropped in the async ripple?

## 7. Companion / boundary
- **Slice-2 = TIP-84B (hashed persistent store)** = the security work: a production `IApiKeyStore` impl (DB/hashed-file), keys stored hash-only, loaded via `SecretRefResolver` (TIP-83B pattern) + a `ConfigureEvidenceSigning`-style config-selected backend + a production `no-plaintext-key` startup guard. **GATED on a TIP-15 STOP/RRI decision** (hashing scheme + secret-store posture) - a small decision-brief resolving TIP-15's open items should precede TIP-84B.
- **Slice-3 = TIP-84C (lifecycle)** = issuance/rotation/revocation + an `OperatorAdmin` admin surface (the enum slot exists, nothing consumes it) - largest, most independent, defer.
- This slice = the enabling seam ONLY; it unblocks Slice-2 without making any production-security or policy decision. Preserves TIP-12/13; honors TIP-15; does not touch the D-P0-1 signing-key domain.

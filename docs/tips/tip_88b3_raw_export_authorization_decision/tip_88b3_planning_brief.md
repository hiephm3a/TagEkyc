## 0. Grounding (on-code, CC-verified @ 2c684cf)
- **Session (recon RESOLVED):** `VerificationSession` = `Id` (Guid = EkycSessionId), `ClientApplicationId` (Guid = ownership; NO tenant column), `SubjectRef` (string, required), `ExternalSessionId` (nullable, audit-echo only), `State`. Exportable = `Completed` ONLY. A `WithState` path exists (`VerificationSession.cs:145`) -> `Completed` NOT proven terminal-immutable -> B3 locks + re-checks the session row (2.3).
- **88B1 control plane (consume, do not duplicate/change):** `ResolveExportEligibilityForAuthorization(principalId, policyId, policyVersion)` -> `{State, PrimaryCause, Causes[], EvaluatedAtUtc, BoundRuleSetVersion, CurrentRuleSetVersion, GrantRef, LifecycleRef, FulfillmentRefs[]}` - LOCK-CAPABLE, REQUIRES ambient txn, DB clock. B1 owns grant/lifecycle/fulfillment/activation. `ConsentArtifact` EXCLUDED from B1 -> TIP-88B2 owns subject consent; B3 CONSUMES it. **B1 fulfillment-expiry handoff = a recon gate (2.6 / OQ-B3.8).**
- **88A policy CLIENT-AGNOSTIC:** grant is principal->policy. Exact allowed classes = `raw_export_policy_allowed_classes (PolicyId, PolicyVersion, RawClass)` = `PolicyAllowedClasses`.
- **TIP-88B2 consent resolver (consume the FINAL 3-parameter signature ONLY):** `ResolveSubjectExportConsentForAuthorization(verificationSessionId, policyId, policyVersion)` - **B3 passes its `ResolvedVerificationSessionId` (asserted non-null) as `verificationSessionId`, NEVER the requested id**, and passes NO purpose/recipient/requiredRawClasses/SubjectRef; the resolver DERIVES `PurposeCode` (Slice-1 constant) and `RecipientClientApplicationId` (locked session owner) itself. Returns `{Effective, ConsentedRawClasses[], Revision, SubjectConsentRecordId, ConsentScopeHash, PurposeCode, RecipientClientApplicationId, ValidFromUtc, ValidUntilUtc, SubjectRef, Provenance{...}, Cause(missing|withdrawn|expired)}`. **B3 BINDS the RETURNED `PurposeCode`, `RecipientClientApplicationId`, `ConsentScopeHash`, `SubjectConsentRecordId`, `Revision`, and validity snapshot into the decision evidence (2.5)** - it does not compute them independently. The resolver emits a FACTUAL consent-state `Cause`, NOT an authorization denial cause - B3 maps it (2.7 / B5).
- **Identity:** `AuthenticatedClientContext` carries `PrincipalId` (Guid) + `ClientApplicationId` + `AllowedClientApplicationIds` + `ApiKeyId` (credential/audit id).

## 1. Intent
Per raw-export ATTEMPT: an idempotent, evidence-bound, revalidatable authorization DECISION over 88B1 + the TIP-88B2 subject consent + a bound verification session - WITHOUT raw bytes. B3 OWNS class-selection (Option-B, 2.8), policy-ceiling + consent-coverage checks, and the deterministic cause taxonomy (2.7). **B3 does NOT own authoritative purpose/recipient derivation - TIP-88B2 owns that; B3 owns strict same-owner validation, the assertion that the recipient RETURNED by the B2 resolver matches the locked session owner and the caller, and BINDING the returned values into decision + permit evidence (2.2).** Only on Authorize does it freeze a metadata permit the assembly slice REVALIDATES + consumes.

## 2. Scope - ALLOWED
### 2.1 Verification-session binding (STRICT same-owner) + requested-vs-resolved identity
- **[FINAL - requested vs resolved session identity]** a decision row carries TWO distinct session columns, because a `SESSION_NOT_FOUND` decision must still be durable + idempotent while binding NO session FK:
  - **`RequestedVerificationSessionId`** - NON-NULL, **no FK** (it may name a session that does not exist), and it is the value used for the **idempotency key/fingerprint** (2.4).
  - **`ResolvedVerificationSessionId`** - NULLABLE **FK** to the authoritative session row.
  - **`SESSION_NOT_FOUND`:** requested PRESENT; `ResolvedVerificationSessionId` NULL; **ALL** authoritative snapshot columns (owner, `SubjectRef`, session version/state) NULL. No row existed, so nothing can be snapshotted.
  - **`SESSION_NOT_OWNED` (session EXISTS but belongs to another client) - LOCKED asymmetry:** `ResolvedVerificationSessionId` **POPULATED** (the row was found + locked), session **owner `ClientApplicationId` POPULATED (internal-evidence only)**, session **version/state POPULATED**, but **`SubjectRef` MUST be NULL.**
    - *Rationale, locked:* the owner id is needed to PROVE why the attempt was denied (owner mismatch) and is a CLIENT identifier, not subject PII; `SubjectRef` identifies another client's DATA SUBJECT and would let a caller accumulate cross-client subject identifiers simply by probing session ids. So: **owner persisted internal-only; subject never persisted.** Public response stays normalized with `SESSION_NOT_FOUND` (2.7) - the internal/public split is what makes persisting the owner safe.
  - **Owned session (all later gates):** `ResolvedVerificationSessionId == RequestedVerificationSessionId` AND the FULL authoritative snapshot (owner, `SubjectRef`, version/state) is POPULATED from the locked row.
  - **An `Authorized` decision and its permit REQUIRE `ResolvedVerificationSessionId` NON-NULL**; the permit binds ONLY the authoritative RESOLVED session (never the raw requested id).
- `EkycSessionId` = `VerificationSession.Id` (Guid). Gates: session EXISTS; `State == Completed`; **STRICT EQUALITY `session.ClientApplicationId == AuthenticatedClientContext.ClientApplicationId`, else DENY.** `AllowedClientApplicationIds` is a GENERAL access allowlist (`ApplicationAuthorization.cs:65`), NOT raw-BIO delegation - **NOT used here** (future OQ-B3.6). `ExternalSessionId` = audit echo only. Load-bearing: the client-agnostic policy grant makes this the cross-client guard.

### 2.2 Purpose + recipient VALIDATION/BINDING (derivation is TIP-88B2's) + subject consent (CONSUMED)
- **Never accepted from the authorization request.** For the same-owner CHECK B3 takes the caller's `AuthenticatedClientContext.ClientApplicationId` and VERIFIES it equals the locked session owner (2.1, 2.3 step 5). Same-owner Slice-1 only.
- **`PurposeCode`** = the Slice-1 constant `SubjectRawBiometricExport`, **derived authoritatively by TIP-88B2** (its 2.1a); NOT request input.
- **[B3-4 - ownership split, exact]** **TIP-88B2 OWNS the authoritative derivation of `PurposeCode` + `RecipientClientApplicationId` for the consent scope** (from the locked session row). **TIP-88B3 OWNS:** (a) STRICT same-owner validation of the caller against the locked session; (b) the ASSERTION that the recipient RETURNED by the TIP-88B2 resolver matches the locked session owner AND the caller's `ClientApplicationId` (mismatch => fail closed); (c) BINDING the returned `PurposeCode` + `RecipientClientApplicationId` into the decision evidence AND the permit. B3 does NOT re-derive the consent scope; the returned values are authoritative.
- **Subject consent CONSUMED from TIP-88B2** (record model + authority + validity + `ConsentScopeHash` + purpose/recipient derivation are B2). B3 calls the 3-parameter `ResolveSubjectExportConsentForAuthorization(ResolvedVerificationSessionId, policyId, policyVersion)` inside its txn (resolved id only, asserted non-null first - 2.3 step 10), requires `Effective` (missing/withdrawn/expired -> DENY, 2.7/B5), reads `ConsentedRawClasses[]`, and does the coverage check. NO re-modeling of consent.

### 2.3 Authorization transaction order (pinned; one atomic transaction)
After **syntactic request validation** (2.8; a validation failure makes NO idempotency claim + NO durable decision), in exactly this order:
1. begin transaction;
2. claim idempotency (2.4);
3. **ATTEMPT to load + lock the `VerificationSession` row for `RequestedVerificationSessionId`. If it does NOT exist there is no row to lock** -> write an immutable `Denied` `SESSION_NOT_FOUND` decision (requested PRESENT, `ResolvedVerificationSessionId` NULL, session snapshot NULL, **NO 88B1 call, NO TIP-88B2 call, NO permit**, zero `AuthorizedRawClasses`, `DecisionExpiresAtUtc` NULL) and COMMIT - the remaining steps are SKIPPED;
4. (session exists) load/bind authoritative owner, `SubjectRef`, session version/`State`; set `ResolvedVerificationSessionId = RequestedVerificationSessionId`;
5. require STRICT same-owner (derive `RecipientClientApplicationId` = caller client, verify == locked owner) AND `State == Completed`;
6. call the TIP-88B1 resolver (ambient txn; holds B1 shared locks) -> require `State == Active`;
7. load exact 88A `PolicyAllowedClasses` for `(PolicyId, PolicyVersion)`;
8. derive `EffectiveRequestedRawClasses` (Option-B, 2.8);
9. policy ceiling `EffectiveRequestedRawClasses ⊆ PolicyAllowedClasses` (else Denied `REQUESTED_RAW_CLASSES_NOT_ALLOWED`);
10. **ASSERT `ResolvedVerificationSessionId` is NON-NULL**, then call the TIP-88B2 consent resolver **`(ResolvedVerificationSessionId, policyId, policyVersion)`** - the RESOLVED id ONLY, never `RequestedVerificationSessionId` (a requested id may name a non-existent session and must never be used to resolve consent) - (ambient txn; holds consent-scope shared lock) -> require `Effective`; bind the RETURNED purpose/recipient/scope-hash/record-id/revision/validity;
11. consent coverage `EffectiveRequestedRawClasses ⊆ ConsentedRawClasses` (else Denied `SUBJECT_CONSENT_CLASS_COVERAGE_INSUFFICIENT`);
12. write immutable decision + evidence (2.5);
13. write permit ONLY for `Authorized` (2.6);
14. commit.

**All resolver locks (B1 shared + TIP-88B2 consent-scope + session row) remain HELD until commit/rollback.** **[B3-3] Two-connection race tests owned by B3:** authorize racing **88B1 grant / lifecycle / fulfillment mutation**, **consent-withdraw**, **session-transition**, and **duplicate authorization** -> never a stale `Authorized`; concurrent duplicate -> exactly one decision. **Authority-revoke races are NOT a B3 test obligation** - they belong to the TIP-88B2 consent-COMMAND tests (an authority governs who may WRITE consent, not whether an already-effective consent reads valid at authorization time). **[B3-5] There is NO runtime rule-set publisher** (88A rule tables are migration-only; the landed reject-trigger enforces it). A rule-publish race is therefore **NOT** a B3 test obligation and is only relevant IF a publisher is ever introduced by a future TIP - **do not reintroduce a publisher to satisfy a test.**

### 2.4 Idempotency + replay [DEDICATED table; PATTERN not the session-scoped boundary]
- **Idempotency key: `(PrincipalId, ClientApplicationId, RequestedVerificationSessionId, IdempotencyKey)`** - the REQUESTED id (2.1), never the resolved FK, so a `SESSION_NOT_FOUND` attempt is still idempotent in a DEDICATED append-only `raw_export_authorization_idempotency` (the existing `append_idempotency_records` is `(VerificationSessionId, IdempotencyKey)` only - `RepositoryPorts.cs:69` / `TagEkycDbContext.cs:211` - not reusable). Reuse the PATTERN (TIP-80S).
- **Semantic request fingerprint (exact):** `PrincipalId`, `ClientApplicationId`, `RequestedVerificationSessionId`, `PolicyId`, `PolicyVersion`, `PurposeCode`, `RawClassSelectionMode`, and (when `ExplicitSubset`) canonical `RequestedRawClasses` (2.8).
  - **`RecipientClientApplicationId` is NOT in the fingerprint** - it is derived and invariant-equal to `ClientApplicationId` (already present); it is bound to decision EVIDENCE (2.5), not request identity (B2).
  - **`SubjectRef` NOT in the fingerprint** - server-derived, evidence-bound.
  - **`ApiKeyId` NOT in the fingerprint** - credential evidence only.
- **Replay (LOCKED):** idempotency claimed BEFORE business eval. Same key+fingerprint => SAME immutable decision (Authorized OR Denied); same key+different fingerprint => conflict; concurrent duplicates => exactly ONE decision; re-eval after `Denied` requires a NEW key; a rolled-back infra failure => NO durable decision. `DefaultPolicySet` vs `ExplicitSubset` under one key => conflict.

### 2.5 Decision + evidence (single immutable row per attempt) [dedicated typed table]
- Dedicated append-only `raw_export_authorization_decisions` row per attempt (append-only trigger; NOT audit_events). `Outcome` in `{Authorized, Denied}` (CHECK). Typed columns: `ExportDecisionId`, idempotency fingerprint hash, `PrincipalId`, `ClientApplicationId`, `ApiKeyId` (credential evidence), **`RequestedVerificationSessionId` (NOT NULL, no FK)**, **`ResolvedVerificationSessionId` (nullable FK)**, server-bound `SubjectRef` (NULL when unresolved), `PolicyId`, `PolicyVersion`, bound `RuleSetVersion`, evaluated `GrantRef`/`LifecycleRef`/`FulfillmentRefs[]`, **`PurposeCode`**, **derived `RecipientClientApplicationId`**, **`ConsentScopeHash`**, **`SubjectConsentRecordId`**, **`ConsentRevision`**, **consent validity snapshot (`ConsentValidFromUtc`/`ConsentValidUntilUtc`) used for expiry**, `PrimaryCause` + `Causes[]` (2.7), `DecisionExpiresAtUtc`, `DecidedAtUtc` (DB-stamped). [B1: the decision binds the FULL authorization scope - purpose, derived recipient, scope hash, consent record id + revision + validity snapshot.]
- **[FINAL - per-cause NULLABILITY of session-derived + B2-derived columns]** these columns are NULLABLE and populated ONLY once the producing step actually ran (2.3). A gate that fails BEFORE the TIP-88B2 resolver call CANNOT have B2-derived values, because the resolver was never invoked:
  - **`SESSION_NOT_FOUND`** - `ResolvedVerificationSessionId` NULL; owner / `SubjectRef` / session-version NULL; **`PurposeCode`, `RecipientClientApplicationId`, `ConsentScopeHash`, `SubjectConsentRecordId`, `ConsentRevision`, consent-validity snapshot ALL NULL** (B2 not called); no B1 refs; no permit.
  - **`SESSION_NOT_OWNED`** - resolved session + session-version populated (the row was locked), **but B2 is NOT called** => all B2-derived columns NULL; no B1 refs; no permit. **`SubjectRef` is NOT persisted for a not-owned attempt** (it belongs to another client's subject; withholding it keeps the internal record from accumulating cross-client subject identifiers). If a deployment deliberately wants it for forensics, that must be an EXPLICIT internal-only decision, never surfaced publicly (public response stays normalized with `SESSION_NOT_FOUND`, 2.7).
  - **`SESSION_NOT_COMPLETED`** - resolved session + owner + `SubjectRef` + session-version populated; B2 NOT called => B2-derived columns NULL; no permit.
  - **`EXPORT_ELIGIBILITY_INACTIVE`** - session columns populated + B1 refs/causes populated; B2 NOT called => B2-derived columns NULL.
  - **`REQUESTED_RAW_CLASSES_NOT_ALLOWED`** - session + B1 + policy/class snapshots populated; **B2 NOT called** (ceiling precedes the consent resolver, 2.3 steps 9-10) => B2-derived columns NULL.
  - **`SUBJECT_CONSENT_NOT_EFFECTIVE`** - B2 WAS called: `PurposeCode`, `RecipientClientApplicationId`, `ConsentScopeHash` populated from the resolver; `SubjectConsentRecordId` / `ConsentRevision` / consent-validity may be NULL (no effective revision exists); factual cause persisted (2.7 B5).
  - **`SUBJECT_CONSENT_CLASS_COVERAGE_INSUFFICIENT`** and **`Authorized`** - ALL session + B1 + B2-derived columns populated (`Authorized` additionally requires `ResolvedVerificationSessionId` NOT NULL, non-empty `AuthorizedRawClasses`, one permit, non-null expiry).
- **Typed raw-class snapshots (child rows, 2.8):** `PolicyAllowedClasses`, `RawClassSelectionMode`, `RequestedRawClasses`, `EffectiveRequestedRawClasses`, `AuthorizedRawClasses`.
  - **Authorized:** `AuthorizedRawClasses = EffectiveRequestedRawClasses`, NON-EMPTY.
  - **Denied:** `AuthorizedRawClasses` = ZERO rows, NO permit, `DecisionExpiresAtUtc` NULL (2.6) - **never label unapproved classes authorized.**
- **[B4 - deterministic Denied evidence matrix]** (replaces "as needed"; PrimaryCause = first failing gate in 2.3). `Y` = present, `—` = absent, `norm` = public-normalized:

  | PrimaryCause | Policy Allowed | Requested + Mode | Effective | B1 refs+causes | Consent ref/rev | Consented classes | Auth classes / permit |
  |---|---|---|---|---|---|---|---|
  | SESSION_NOT_FOUND (norm; resolved NULL, snapshot ALL null) | — | Y | (ExplicitSubset only) | — | — | — | 0 / none |
  | SESSION_NOT_OWNED (norm; resolved + owner + version populated, `SubjectRef` NULL) | — | Y | (ExplicitSubset only) | — | — | — | 0 / none |
  | SESSION_NOT_COMPLETED | — | Y | (ExplicitSubset only) | — | — | — | 0 / none |
  | EXPORT_ELIGIBILITY_INACTIVE | — | Y | (ExplicitSubset only) | Y | — | — | 0 / none |
  | REQUESTED_RAW_CLASSES_NOT_ALLOWED | Y | Y | Y | Y | — | — | 0 / none |
  | SUBJECT_CONSENT_NOT_EFFECTIVE | Y | Y | Y | Y | factual cause (rec/rev may be null) | — | 0 / none |
  | SUBJECT_CONSENT_CLASS_COVERAGE_INSUFFICIENT | Y | Y | Y | Y | Y | Y | 0 / none |
  | Authorized | Y | Y | Y | Y | Y | Y | non-empty / ONE permit |

  **Session-identity rule, PER CAUSE (2.1):** `SESSION_NOT_FOUND` => resolved NULL + owner/`SubjectRef`/session-version ALL NULL. `SESSION_NOT_OWNED` => resolved POPULATED + owner POPULATED (internal-evidence only) + session-version POPULATED, but **`SubjectRef` NULL**. `SESSION_NOT_COMPLETED` and every later cause (incl. `Authorized`) => resolved == requested with the FULL snapshot (owner, `SubjectRef`, version/state) populated. `Authorized` and its permit additionally REQUIRE resolved NON-NULL.
  **Rule:** a typed `EffectiveRequestedRawClasses` snapshot is PRESENT for EVERY syntactically-valid attempt that REACHES policy evaluation (step 8+); for pre-policy failures it is present only when `ExplicitSubset` (derivable without the policy set), and `RawClassSelectionMode` + `RequestedRawClasses` are ALWAYS recorded. Every `Denied` => zero `AuthorizedRawClasses` + no permit.
- Written in the SAME transaction as the eligibility + consent reads (atomic). Only `Authorized` yields a permit.

### 2.6 Permit persistence + expiry (metadata-only; revalidated, NOT bearer)
- **Dedicated `raw_export_authorization_permits`:** `PermitId` PK, `AuthorizationDecisionId` UNIQUE FK, EXACTLY ONE permit per `Authorized`, NONE for `Denied`. Decision + permit + child rows in ONE transaction; append-only (no later child insert/update/delete).
- **[B1 - LOCKED] the permit DIRECTLY FREEZES `PurposeCode` + `RecipientClientApplicationId`** (the values returned by the TIP-88B2 resolver). There is NO "join the decision instead" alternative: the assembly slice reads them from the permit and enforces them at consume time (it MAY additionally join the decision evidence, but the permit is self-contained for purpose/recipient).
- **Permit fields:** `PermitId`, `AuthorizationDecisionId`, **`ResolvedVerificationSessionId` (NOT NULL FK - the permit binds ONLY the authoritative resolved session, never the raw requested id, 2.1)**, `SubjectRef`, `PolicyId`, `PolicyVersion`, `PurposeCode`, `RecipientClientApplicationId`, `DecisionExpiresAtUtc`, `SchemaVersion`. **Freezes ONLY `AuthorizedRawClasses`.** NO bytes/hash/manifest.
- **Expiry (LOCKED):** `DecisionExpiresAtUtc = min(EvaluatedAtUtc + PermitTtl, Consent.ValidUntilUtc ?? MaxValue, earliest(Fulfillment.ValidUntilUtc ?? MaxValue))`, DB time; NO caller TTL.
- **[B3-2 - expiry presence semantics]** for an **`Authorized`** decision `DecisionExpiresAtUtc` is **REQUIRED (NOT NULL)** and **EQUALS the permit's expiry** (one value, no divergence - CHECK/test asserted). For a **`Denied`** decision `DecisionExpiresAtUtc` is **NULL**, there is **NO permit**, and `AuthorizedRawClasses` has **ZERO child rows** (a Denied decision carries retention/evidence metadata only, never a usable expiry).
- **[B6 - B1 expiry handoff, recon gate OQ-B3.8]** recon the LANDED 88B1 resolver return before build. If fulfillment `ValidUntilUtc` is absent, add an ADDITIVE 88B1 projection (`FulfillmentRefs[].ValidUntilUtc` or `EarliestFulfillmentValidUntilUtc`); **B3 MUST NOT re-query/re-derive B1 fulfillment outside the resolver.** Pin `PermitTtl` CONFIG SOURCE (key), MIN/MAX BOUNDS, PRODUCTION READINESS FAILURE behavior if unset/out-of-range, and DB transaction-time semantics.
- **Not a bearer token:** a non-consumed permit is UNUSABLE if, before assembly, the grant is revoked / policy Suspended-Revoked / bound rule set stale / a fulfillment withdrawn-expired / the subject consent withdrawn-expired / the permit expires. The assembly slice revalidates all gates + expiry + one-time consumption. **B3 persists ONLY decision/permit METADATA; NO consumption ledger** (assembly owns it).
- **[B3-2 - ASSEMBLY HANDOFF REQUIREMENT, normative for the future assembly slice]** BEFORE consuming a permit, the assembly slice MUST revalidate, against the freshly LOCKED session row: (1) the `VerificationSession` still EXISTS; (2) `session.ClientApplicationId == Permit.RecipientClientApplicationId`; (3) `session.SubjectRef == Permit.SubjectRef`; (4) `session.State == Completed`. **A permit is UNUSABLE if the session is no longer `Completed`** (a session that left `Completed` after the decision invalidates the permit), or if owner/subject no longer match. This is in ADDITION to revalidating grant / lifecycle / fulfillment / rule-set / subject-consent / expiry and the one-time-consumption ledger. (Note the asymmetry with TIP-88B2: consent WITHDRAWAL is deliberately NOT gated on `Completed`, but export CONSUMPTION is.)

### 2.7 Cause taxonomy (B3-owned, deterministic)
- **PERSISTED `PrimaryCause` taxonomy** (PrimaryCause = first failing gate in 2.3), complete and ordered: (1) `SESSION_NOT_FOUND`; (2) `SESSION_NOT_OWNED`; (3) `SESSION_NOT_COMPLETED`; (4) `EXPORT_ELIGIBILITY_INACTIVE` (carries 88B1 `PrimaryCause`/`Causes[]`); (5) `REQUESTED_RAW_CLASSES_NOT_ALLOWED`; (6) `SUBJECT_CONSENT_NOT_EFFECTIVE`; (7) `SUBJECT_CONSENT_CLASS_COVERAGE_INSUFFICIENT`.
- **[B3-3] `REQUEST_VALIDATION_FAILED` is NOT part of the persisted `PrimaryCause` taxonomy.** Syntactic request validation (2.8) is a PREFLIGHT error returned to the caller BEFORE the transaction: it creates NO durable decision, NO idempotency claim, and therefore NEVER appears as a persisted `PrimaryCause`.
- **[B5 - preserve exact TIP-88B2 factual cause] for non-effective consent:** `PrimaryCause = SUBJECT_CONSENT_NOT_EFFECTIVE`, and `Causes[]` INCLUDES the EXACT TIP-88B2 factual cause (`missing` / `withdrawn` / `expired`). Internal evidence retains the exact facts; the PUBLIC response MAY normalize.
- **Anti-enumeration:** `SESSION_NOT_FOUND` and `SESSION_NOT_OWNED` return an IDENTICAL opaque public "not found / not accessible" (no signal distinguishing another-client's session from a non-existent one). Internal evidence keeps the exact `PrimaryCause`.

### 2.8 Raw-class request semantics (Option B, LOCKED - Homeowner)
- `RequestedRawClasses` OPTIONAL. Omitted/JSON-null (same) => `RawClassSelectionMode = DefaultPolicySet`; `EffectiveRequestedRawClasses = PolicyAllowedClasses`. Supplied => `ExplicitSubset`; read as raw ORDERED list; non-empty; map to CLOSED taxonomy; canonical-normalize; REJECT duplicate canonical BEFORE sorting; stable-sort => canonical `EffectiveRequestedRawClasses`.
- **Explicit-list validation BEFORE the idempotency claim.** Validation failures (malformed/unknown/empty/duplicate) => NO durable decision, NO idempotency claim.
- **Deterministic order:** policy ceiling BEFORE consent coverage.
- **Idempotency binds `RawClassSelectionMode` + (ExplicitSubset) canonical `RequestedRawClasses`** (2.4). DefaultPolicySet vs ExplicitSubset => conflict; an explicit full set is DISTINCT from omitted default.
- **Permit freezes ONLY `AuthorizedRawClasses`** (= PolicyAllowedClasses for DefaultPolicySet, = canonical RequestedRawClasses for ExplicitSubset).

### 2.9 DB privilege / SECURITY DEFINER function security (locked)
- For the idempotency, decision, permit, and typed child tables: **runtime role has NO direct `INSERT`/`UPDATE`/`DELETE`/`TRUNCATE`**; it ONLY `EXECUTE`s the intended command function(s). **`PUBLIC` EXECUTE revoked.** SECURITY DEFINER manifest: expected NON-LOGIN owner + FIXED `search_path` + no unsafe dynamic SQL. **Readiness FAILS CLOSED** on ACL/owner/`search_path`/`prosecdef`/signature drift. **Direct-SQL bypass tests** (runtime cannot mutate these tables directly). **Same-transaction parent/child closure** (decision + permit + all class children in one txn); **no later child insertion/update/delete** (xmin same-txn guard + append-only triggers).

## 7. Review tier & attacks
Tier-1. Attacks: (a) cross-client session export or session ENUMERATION via distinguishable not-found/not-owned (2.7)? (b) export over non-Completed, or a session that left `Completed` after eligibility (2.3 re-check)? (c) caller-injected purpose/recipient, recipient wrongly in the request fingerprint, or B3 re-deriving the consent scope instead of binding the TIP-88B2-returned values (2.2/2.4)? (c2) a permit consumed against a session that left `Completed`, changed owner, or changed subject (assembly handoff, 2.6 B3-2)? (d) a permit used after grant/policy/fulfillment/consent revocation (bearer bug)? (e) stale `Authorized` after a concurrent grant-revoke / consent-withdraw / session-transition (locks must hold to commit; rule-set publish is NOT applicable - no runtime publisher exists, 2.3)? (e2) a `SESSION_NOT_FOUND` path that tries to lock a non-existent row, bind a resolved FK, or call B1/B2 (2.3 step 3, 2.5 nullability)? (f) duplicate/replayed authorize -> two decisions? (g) B3 duplicates/bypasses the B1 resolver or re-queries B1 fulfillment outside it (2.6)? (h) a Denied row with non-empty `AuthorizedRawClasses` or a permit, or an evidence field present/absent violating the matrix (2.5/B4)? (i) runtime direct-DML on B3 tables / PUBLIC EXECUTE (2.9)? (j) any raw-byte read/assemble/return path?

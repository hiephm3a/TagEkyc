NO `app.UseAuthentication/UseAuthorization`, NO `[AllowAnonymous]`/`RequireAuthorization` anywhere. - **Redline surface:** the validators return ONLY opaque codes - never connection strings, table contents, PII, or key material. `/readiness` echoing ONLY those codes is safe; the one hazard is dumping an exception `.Message` verbatim (if messages ever get enriched) or echoing the resolved connection string.

## 1. Intent
Give hospital IT / the reverse proxy a real **`/readiness`** probe: a read-only endpoint that reports whether the critical production dependencies are actually ready (DB reachable + migrated + required tables; API-key store ready; signer/JWKS ready; production config posture), returning `200 ready` / `503 not-ready` with a **codes-only** body. It RE-USES the three existing re-runnable validators - no new checks, no validator-logic change, no migration. `/health` stays a dependency-free liveness probe.

## 2. Scope - ALLOWED

### 2.1 Add `GET /readiness` (minimal-API in `Program.cs`, anonymous by omission)
- New `app.MapGet("/readiness", ...)` next to `/health` (`Program.cs:44`). Anonymous the same way the other probes are (it does NOT call `IApiKeyAuthenticator`) - and it MUST NOT depend on the authenticator, because it reports whether the auth STORE itself is ready (a circular dependency would make "auth down" unreportable).
- Resolves a DI scope and runs the applicable validators (below). Read-only; no state change.

### 2.2 Aggregate the three EXISTING re-runnable validators (no refactor of their logic)
- In Production, run all three and collect the stable failure code from each that throws (do NOT stop at the first - report the FULL set so an operator sees every unready dep):
  - `PostgresProductionReadinessValidator.ValidateAsync` -> `PROD_DB_*` codes.
  - `ApiKeyStoreProductionReadinessValidator.ValidateAsync` -> `PROD_APIKEY_NO_ACTIVE_KEYS` (+ pepper).
  - signer/JWKS `ValidateStartup` -> signing readiness. **[self-check] Honesty caveat:** this reports config-validity + a signer self-test ("can produce a signature with the configured key"), NOT live remote-HSM reachability - for the shipped P12/LocalDev signer that is the full story; a future PKCS11/HSM backend may need a separate liveness probe. Do NOT over-claim `/readiness` as an HSM-up check.
- **[R:Codex-3/GPT-P1-04] signer/JWKS code mapping - ALLOWLIST, do NOT free-echo `.Message`:** unlike the DB/api-key validators (typed exceptions with a stable `.Code`), the signer/JWKS path throws `InvalidOperationException` whose MESSAGE is a stable code like `PROD_SIGNING_*`/`JWKS_*` (`Es256JwksProvider.cs:61`). Mapping rule: if the caught message EXACTLY matches a member of a small KNOWN ALLOWLIST of stable signer/JWKS codes -> surface that code; ANY other/unrecognized message -> generic `PROD_READINESS_CHECK_FAILED` (never echo the raw message). This avoids leaking an arbitrary `InvalidOperationException.Message` while preserving the known codes.
- **[R:Codex-2/4 + GPT-P1-02 - OQ-2 LOCKED] include production posture gates, read ALREADY-BOUND options only:** assert `Persistence:Provider == Postgres`, `ApiKeyStore:Backend == Postgres`, evidence-signer backend `!= LocalDev`.
  - **[R2-Codex-2 + R3-Codex-2 - PINNED] register `IOptions<>` read-only:** only `EvidenceSigningOptions` is registered via `AddOptions` (`Program.cs:227`); `TagEkycPersistenceOptions` + `ApiKeyStoreOptions` are currently bound LOCALLY inside `ConfigurePersistence`/`ConfigureApiKeyStore` (`:74`/`:124`) and are NOT registered as `IOptions<>`. DECISION (not builder's choice): the build **registers `IOptions<TagEkycPersistenceOptions>` + `IOptions<ApiKeyStoreOptions>` READ-ONLY** in `Program.cs` (additive, no behavior change, consistent with `EvidenceSigningOptions`); the posture check consumes those. Do NOT use the ad-hoc "bind once inside the check" path (it diverges code vs test).
  - **[R2-Codex-3 + R3-Codex-minor - posture codes, reuse-where-exists else generic]:** LocalDev-signing => reuse `PROD_SIGNING_LOCALDEV_BACKEND_FORBIDDEN` (`Program.cs:296`). `RequireHardwareSigner=true` with a non-Pkcs11 backend throws only a GENERIC message (`:230`/`:281`, NO stable code) => map to generic `PROD_READINESS_CHECK_FAILED` (do NOT mint a signing code here). Persistence: in-memory => reuse `PROD_PERSISTENCE_INMEMORY_FORBIDDEN` (the exact stable code); an unsupported/other provider (generic message today, no stable code) => generic `PROD_READINESS_CHECK_FAILED`. Api-key backend != Postgres => reuse `PROD_APIKEY_LOCALDEV_STORE_FORBIDDEN`. Read the SAME bound option the startup path reads; do NOT re-derive signer/persistence selection logic.
- **[R2-Codex-1] the endpoint aggregates via a small WRAPPER SEAM, NOT by refactoring the validators:** add a thin `IReadinessCheck` (one wrapper per dependency) - each wrapper CALLS the existing concrete validator's `ValidateAsync`/`ValidateStartup` and maps its outcome to a code; the endpoint depends on the set of `IReadinessCheck`. The concrete validators + hosted services are UNCHANGED (no logic touched). Tests inject FAKE `IReadinessCheck` implementations into the endpoint's set. This is the "DI-resolved abstraction" - it is a NEW thin wrapper, not a validator refactor.

### 2.3 Response contract - CODES ONLY (the leak redline)
- All pass => `200` `{ "status": "ready" }`.
- Any fail => `503` `{ "status": "not-ready", "codes": ["PROD_DB_UNREACHABLE", ...] }` - the stable codes ONLY.
- **[R:Codex-5/GPT-P1-05] `codes` are DISTINCT + DETERMINISTIC order** by check group: (1) posture, (2) database, (3) api-key store, (4) signer/JWKS, (5) unexpected-generic. No duplicates. (Stable order = stable tests + easier operator reading.)
- **NEVER** put in the body: a resolved connection string, an api-key value, PII, a raw exception `.Message`, a table's contents, or a private filesystem path. Map each caught TYPED readiness exception to its stable `.Code`, not its message.
- **[self-check] Untyped/unexpected exception => fail-closed WITHOUT leaking:** if a validator throws something that is NOT a typed readiness exception with a `.Code` (a raw `NpgsqlException`/`TimeoutException`/`NullReferenceException` - whose `.Message` can carry the host/connection detail), the endpoint records a GENERIC code (e.g. `PROD_READINESS_CHECK_FAILED`) and returns `503` - it MUST NOT surface the raw `.Message`. This is the main leak vector: an unwrapped DB error message could contain the host/credentials.
- **[R:GPT-P1-06] exact headers:** `Cache-Control: no-store` + `Pragma: no-cache`.
- **[R:GPT-P1-07 + R2-Codex-4] bounded/cancellation - honest about sync vs async:** the timeout is GUARANTEED for the ASYNC checks (DB + api-key `ValidateAsync` take a `CancellationToken`): the endpoint passes the request CT + a bounded linked-CTS so a hung/slow DB does not make `/readiness` hang; timeout => `503` + generic `PROD_READINESS_CHECK_FAILED`, NO raw message. The signer/JWKS `ValidateStartup` is SYNCHRONOUS `void` (`Es256JwksProvider.cs:19`) - it is bounded only insofar as the CURRENT impl is local/cheap; a linked-CTS cannot interrupt a sync call if a future HSM blocks. A future HSM liveness probe is a SEPARATE concern (out of scope) - do NOT over-claim a hard timeout on the sync signer check.

### 2.4 Non-Production behavior (LOCKED, OQ-3)
- In non-Production the three prod hosted-services/validators are not wired. `/readiness` returns `200 {status:"ready"}` IMMEDIATELY in non-Production (Dev does not enforce prod deps; the probe exists in all envs for uniform monitoring wiring). It does NOT resolve/run the prod checks in non-prod - immediate ready, same minimal body/contract.

### 2.5 [R:Codex-1 - the load-bearing semantics] Runtime-degradation, NOT boot-time (relationship to startup fail-fast)
- The three prod hosted services **fail-fast at startup** (`StartAsync` throws -> host aborts) BEFORE Kestrel accepts requests (`Program.cs:91`/`:166`/`:270`). So if DB/key/signing is broken AT BOOT, the app never starts and `/readiness` is simply unreachable - a "boot-time DB down => `/readiness` 503" claim is IMPOSSIBLE and must NOT be a DoD.
- `/readiness`'s real job = detect **RUNTIME degradation of a dependency that was healthy at boot** (e.g. the DB connection drops mid-service): the monitoring probe then returns `503` so the reverse proxy pulls the instance out of rotation. Frame the endpoint + tests around this, not boot-time failure.
- **Test seam = the `IReadinessCheck` wrappers (§2.2), not the validators.** Tests drive the `503` paths hermetically - without a real DB - via a `WebApplicationFactory` that substitutes the endpoint's `IReadinessCheck` set with fakes.
- **[R3-Codex-1 - PINNED test-host recipe] the fail-fast hosted services would otherwise abort a Production test host at boot.** So the test host: (1) runs `Production`; (2) **REMOVES the three readiness `IHostedService` registrations in test-only** (`PostgresProductionReadinessHostedService`/`ApiKeyStoreProductionReadinessHostedService`/`EvidenceSignerStartupValidationHostedService`) so startup does not require real DB/key/signer; (3) substitutes fake `IReadinessCheck` implementations. This exercises the endpoint's `503`/`200` contract in Production semantics WITHOUT real infra and WITHOUT changing production wiring (the removal is test-only). The production app keeps all three hosted services + fail-fast UNCHANGED.

## 5. Review tier & attacks
Tier-2 (a production HTTP surface; low blast-radius but publicly reachable). Attacks: (a) does the `/readiness` body leak a connection string / key / PII / raw exception `.Message` / private path? (b) does `/readiness` depend on `IApiKeyAuthenticator` (circular - cannot report "auth store down")? (c) does it rebuild or alter `/health`//build//jwks instead of only adding `/readiness`? (d) does a DB-down also break `/health` (liveness MUST stay dep-independent)? (e) does it mutate any validator's logic or run a state-changing check? (f) DoS/enumeration: is the check bounded/cheap, and is the anonymous exposure acceptable (OQ-1)? (g) does non-Production behavior fail-open incorrectly or throw? (h) does the brief/impl wrongly claim a BOOT-TIME dep failure yields a `/readiness 503` (impossible - startup fail-fast aborts the app first)? (i) does the signer/JWKS path free-echo an `InvalidOperationException.Message` instead of allowlist-mapping to a known code or generic? (j) does a hung/slow DB make `/readiness` hang with no bounded timeout? (k) do posture gates re-derive signer/persistence selection logic instead of reading the already-bound options?

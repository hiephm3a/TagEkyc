- **Config pattern to mirror:** `EvidenceSigningOptions.cs:5` (`SectionName` const + flat scalar props) + `.AddOptions<T>().Bind().Validate().ValidateOnStart()` (`Program.cs:249-253`); prod fail-closed via `ValidateProductionTrialP12Configuration` (`Program.cs:318`, reads config + throws `InvalidOperationException("PROD_...")` gated on `IsProduction()`). Stable `PROD_*` family: `PROD_PERSISTENCE_INMEMORY_FORBIDDEN`/`PROD_APIKEY_*`/`PROD_DB_*`/`PROD_SIGNING_*`. **`PROD_RETENTION_*` = greenfield (0 matches).**
- **All existing options are FLAT scalar** (no keyed/per-enum options class in `src`). A `RetentionOptions` keyed per `RetentionClass` would be the first - a small greenfield shape choice (OQ-3).
- **[TENSION - the deferred crux] purge is STRUCTURALLY INCOMPATIBLE with the append-only posture.** The 6 evidence/audit tables (`capture_artifacts`/`evidence_results`/`verification_decisions`/`evidence_packages`/`evidence_manifests`/`audit_events`) are DB-trigger-protected against UPDATE **and** DELETE (`InitialPostgresPersistence.cs:284-319`, `RAISE EXCEPTION`) AND repo-locked to no-mutation by TIP-83E-2's ArchTest. A purge `DELETE` hits the trigger and fails **for any role** unless the trigger is dropped/disabled. So purge needs a DIFFERENT mechanism (partition-drop / a privileged purge role that disables the trigger / retention at the backup/media layer) - an architecture decision touching the evidentiary-integrity guarantee (append-only PROTECTS proof immutability). This must be resolved BEFORE any evaluator or purge job (OQ-1).
- **No recurring background-job infra** (grep `BackgroundService`/`PeriodicTimer`/`Timer` = only one-shot startup `IHostedService`s). A purge JOB = net-new hosting infra (deferred).
- **Raw-biometric retention is OUT** (`verify_and_discard`; raw vault = TIP-82R only, `tip_83_planning_brief.md:24`).

## 1. Intent
Ship the retention CONFIG plumbing: a `RetentionOptions` bound from `TagEkyc:Retention` + a production fail-closed startup guard that throws if a retention policy is absent/invalid - so a prod deployment CANNOT boot without a declared, valid retention window (the guard CREATES the Legal gate). The actual window VALUES are Legal/DPO config input (NOT hardcoded). This slice does NOT classify sessions as `RegulatedEvidence`, does NOT read the dormant tags, does NOT evaluate eligibility, and does NOT purge - those are deferred behind the append-only-vs-purge mechanism decision (OQ-1).

### 2.3 Production fail-closed guard (the deliverable)
- **[R:Codex-1/GPT-P1-04 - RAW-CONFIG guard, not just Options binding]** a plain `.AddOptions<int?>().ValidateOnStart()` would let a non-integer value (`"abc"`) surface as a generic binder/options exception, NOT the stable code. So add a `ValidateRetentionConfiguration(builder)` that reads the RAW config STRING (`builder.Configuration["TagEkyc:Retention:RegulatedEvidenceRetentionDays"]`) and throws `InvalidOperationException(exact stable code)` - mirror `ValidateProductionTrialP12Configuration` (`Program.cs:318`). Gated on `builder.Environment.IsProduction()`:
  - absent/empty => `PROD_RETENTION_POLICY_MISSING`.
  - present but not a parseable positive integer, OR `<= 0`, OR **`> MaxRegulatedEvidenceRetentionDays` (a NAMED technical-sanity ceiling const = `36500` ~100y to catch typos like `999999999` - NOT a Legal value)** => `PROD_RETENTION_WINDOW_INVALID`.
  - the thrown message is the stable CODE only - NO raw binder/options exception text, NO config value echoed.
- The `RetentionOptions` binding (§2.2) still exists for typed consumption; the raw-config guard is what GUARANTEES the stable code. (If instead the options-validation path is used, its exception MUST be caught + re-thrown as the stable code - do not leak the raw binder message.)
- **[R:Codex-P2 - startup ORDERING] run the retention guard AFTER the existing critical config guards** (`ConfigureEvidenceSigning` -> `ConfigurePersistence` -> `ConfigureApiKeyStore` -> retention) so a broken retention config does NOT MASK a `PROD_SIGNING_*`/`PROD_DB_*`/`PROD_APIKEY_*` error when several configs are wrong at once.
- **Non-Production: NO throw** (pass-through, matching how every other `PROD_*` guard is `IsProduction()`-gated). Dev/test boot without retention config.
- The guard forces a prod operator to declare a valid retention window (supplied by Legal) before boot - even though enforcement (classification/eligibility/purge) is deferred, the POLICY DECLARATION gate is meaningful (prevents shipping to prod with no retention policy at all).
- **[self-check - honest framing, NOT theater]** this gate is a policy DECLARATION, NOT enforcement - it does not delete or classify anything. Its concrete value: (a) puts the trial's retention policy ON RECORD + validated before boot; (b) forces the Legal/DPO conversation NOW (a real-patient deployment should declare its retention basis even before automated purge exists - the declared policy can be honored operationally/manually in the interim); (c) gives the deferred enforcement a validated value to consume when built. It is DECOUPLED from OQ-1 (the purge-mechanism fork) - the config gate does not depend on and does not pre-commit that decision. If the Homeowner judges even the declaration gate premature for the trial, the honest fallback is to defer 83E-3 entirely (OQ-4); this brief argues the cheap declaration gate is worth shipping.

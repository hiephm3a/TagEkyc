## 1. Intent
Stand up a PRODUCTION hashed API-key store behind the TIP-84A `IApiKeyStore` seam: a Postgres `ApiKeyRow` (keyed-hash + non-secret prefix, NO plaintext), a `PostgresHashedApiKeyStore` that verifies by prefix-select + constant-time HMAC, a config-selected backend (LocalDev seeds vs Postgres), the pepper loaded via `SecretRefResolver`, and a production fail-closed guard (no LocalDev store / no plaintext pepper in prod). The validator + authorization layer are UNCHANGED (TIP-84A seam absorbs the swap). **"No behavior change" = the auth LOGIC + per-key OUTCOMES are identical** (same codes/order); the intended posture change is only the key SOURCE - production now reads the provisioned Postgres store instead of dev seeds (LocalDev seeds stay for dev/test). Full credential lifecycle (issuance/rotation/revocation admin surface) is TIP-84C; this slice ships only a MINIMAL one-shot provisioning tool so the store is usable for the trial.

### 2.2 `PostgresHashedApiKeyStore : IApiKeyStore`
- **[R:P1-03/P2-8] Key GRAMMAR pinned exactly** (so the builder does NOT invent an incompatible format):
  - Format: `tek_<prefix>_<secret>`.
  - `tek_` = fixed literal scheme tag.
  - `<prefix>` = random NON-SECRET selector, **exactly 16 chars**, alphabet **base64url `[A-Za-z0-9_-]`**; globally UNIQUE via the `KeyPrefix` unique index; stored plaintext + printed. DB `KeyPrefix varchar(16)`.
  - `<secret>` = high-entropy random, **>= 32 random bytes, base64url-encoded (unpadded)**; NEVER stored after the one-shot print.
  - The prefix is the LOOKUP SELECTOR; the HMAC of the FULL presented key is the AUTH.
  - **[R2:6] Verifier-testable secret rule** (the verifier only sees the string, not the byte count): the `<secret>` segment MUST be base64url chars `[A-Za-z0-9_-]` with **length >= 43** (32 bytes unpadded base64url = 43 chars). Shorter/illegal-char => malformed.
  - **Parser:** on any malformed key (wrong scheme tag / wrong segment count / prefix length!=16 / secret length<43 / illegal chars) return `null` (no-match) - **NO exception, NO leak of which check failed.**
- Implements the EXACT seam `Task<ResolvedApiKey?> FindByPresentedKeyAsync(string presentedApiKey, CancellationToken ct)` (no interface change): parse `KeyPrefix` from the presented key -> `SELECT` the `ApiKeyRow` by prefix -> compute `HMAC-SHA256(pepper, presentedApiKey)` -> **`CryptographicOperations.FixedTimeEquals`** vs stored `KeyHash` -> on match, map row -> `ResolvedApiKey` (DROP the hash). `null` on no-prefix-match OR hash-mismatch. `HMACSHA256` + `FixedTimeEquals` are BCL net8.0 (no package; greenfield - no existing repo usage).
- **[R:P1-06] HMAC input encoding byte-exact:** HMAC input = **`UTF8` bytes of the EXACT presented key string** (the full `tek_..._...` as received from the header after extraction). **Do NOT trim/normalize** (trimming creates alternate accepted representations). Whitespace/malformed => parser returns null before the HMAC. The pepper is the HMAC KEY (see §2.5 for its byte encoding). Provisioning and verify MUST use the identical encoding or hashes won't match.
- **[self-check] No-prefix-match timing:** a no-prefix-match may short-circuit to `null` WITHOUT an HMAC - acceptable because the prefix is NON-SECRET (stored plaintext, indexed, printed); it leaks only "this prefix exists", not the secret. (The builder MAY do a dummy `FixedTimeEquals` on no-match to equalize timing; not required since the prefix is non-secret.) The FixedTimeEquals on the HMAC is the load-bearing constant-time check.
- **[R:P1-5 / R2:1 - DI PINNED = scoped auth chain]** the store depends on `TagEkycDbContext` (Scoped). The current `Program.cs` registers `IApiKeyStore` **AND** `LocalDevApiKeyValidator`/`IApiKeyAuthenticator` as **Singleton** (`:25-26`). A singleton validator/authenticator CANNOT consume a scoped store. **DECISION (not builder's choice):** register the WHOLE auth chain - `PostgresHashedApiKeyStore` (or the LocalDev store), `LocalDevApiKeyValidator`, `IApiKeyAuthenticator` - as **Scoped** (they are cheap, per-request; uniform lifetime for both backends keeps DI simple and avoids the captured-DbContext bug). NOT `IDbContextFactory` (rejected: adds a second DbContext lifetime model for no gain here). Add a DI-validation test (build the production service provider, resolve the auth chain, assert no lifetime-scope throw).

### 2.5 Pepper via `SecretRefResolver` (auth-specific codes)
- Resolve the pepper with `SecretRefResolver.Resolve(pepperSecretRef)` (env:/file:), caller-maps-error (TIP-83B pattern `Program.cs:148-167`) to **auth-specific** codes. **[R:P3-13 / R2:4 - taxonomy LOCKED to the TIP-83A `*_SECRET_REF_INVALID`/`*_SECRET_UNAVAILABLE` convention]:**
  - `SecretRefErrorKind.Missing` / empty / unresolved => **`PROD_APIKEY_PEPPER_SECRET_UNAVAILABLE`**
  - `SecretRefErrorKind.Invalid` (malformed ref) OR resolved pepper `< 32 bytes` => **`PROD_APIKEY_PEPPER_SECRET_REF_INVALID`**
  - (Builder does a one-line grep of TIP-83A to confirm the exact suffix spelling and match it; these are NEW codes, not reused, so the grep only confirms the convention - it does NOT reopen the choice.) NOT `PROD_DB_*`/`PROD_SIGNING_*`.
- **[R:P1-05/P2-10] Pepper strength + encoding pinned** (non-empty is NOT enough - a weak pepper makes the keyed-hash theater):
  - **Encoding:** the pepper is treated as **raw UTF-8 bytes** of the resolved string (do NOT base64-decode) - pin this so the HMAC key is reproducible between provisioning + verify.
  - **Minimum strength:** require **>= 32 bytes** (UTF-8) of pepper material; shorter => `PROD_APIKEY_PEPPER_SECRET_REF_INVALID` (fail-closed at startup, same tier as unresolved).
  - **Never log the pepper OR its length as sensitive detail** - surface only the `SourceType`/`RedactedIdentifier` from `SecretRefValue`.
- The resolved pepper stays IN-PROCESS ONLY - never DB, config-plaintext, log, or audit.

### 2.7 Minimal one-shot provisioning (bootstrap only - NOT the 84C admin surface)
- The store has NO writer (issuance = TIP-84C); without a first row every request 401s. Ship a MINIMAL manual one-shot ops tool (a CLI/console command). It is effectively issuance, so its boundary is TIGHT.
- **[R:P1-04] Surface constraints:** NOT registered in the production API; NOT reachable over HTTP; executable ONLY as an ops CLI/manual command; returns a NON-ZERO exit code on any failure; prints the plaintext key EXACTLY ONCE to stdout (operator must capture it into an approved secret store - a documented operational caveat, since a shell that logs stdout would capture it). NO plaintext key to logs/files/DB. **The tool runs in the DEPLOY environment** - it needs the same pepper (via `SecretRefResolver`, raw-UTF8 encoding per §2.5) and DB access as the server so its HMAC matches.
- **[R:P1-3/P1-04] Explicit inputs + policy validation** (do NOT let it mint excessive-scope keys): required inputs `clientApplicationId`, `callerCategory`, `scopes`, optional `expiresAt`. Before insert it MUST load the client policy by `clientApplicationId` (the same `ILocalDevClientPolicyProvider`/configured policy source the validator uses) and REJECT if: policy missing/disabled; requested `scopes` not a subset of `policy.AllowedCallerScopes`; category/scope combo invalid for `BusinessConsumer`/`CaptureAgent`/`TrustedAdapter`; `Allowed*Ids` incompatible with policy.
- **[R:P2-9] Duplicate/concurrent prefix:** on a `KeyPrefix` UNIQUE violation, RETRY prefix generation (small bounded loop, e.g. <=5) - and NEVER print a key until the DB insert SUCCEEDS.
- Row written with `CredentialStatus=Active`, `CredentialType=ManagedApiKey`, `PrincipalId` = the actor principal (DISTINCT from `ApiKeyId`/`CredentialRef`). Full issuance/rotation/revocation + an OperatorAdmin surface = TIP-84C.

### 2.8 [R:#4/P1-09] Policy-provider boundary - EXPLICIT (no production policy store here)
- TIP-84B swaps only the CREDENTIAL store. The validator still fetches the client policy via `ILocalDevClientPolicyProvider` (the `CLIENT_APPLICATION_DISABLED` check reads `policies.Policies`, `LocalDevApiKeyValidator.cs:31`, preserved by TIP-84A). This slice does NOT introduce a production policy store / client-policy lifecycle. **Trial keys bind ONLY to already-configured client policies.** Do NOT claim "full production client-policy lifecycle"; production policy-store neutralization is a separate future slice. A provisioned key for a disabled/absent policy still denies via the policy provider (test below).

### 2.9 [R:#11 / R2:3 - PINNED fail-closed] Empty-table footgun (readiness passes but every request 401s)
- `api_keys` table EXISTS but EMPTY => the service would start and every request 401s (until provisioned). For a real trial that is a footgun. **DECISION (not builder's choice): Production REQUIRES at least one Active API key row at startup.** Zero Active keys in Production => fail-closed `PROD_APIKEY_NO_ACTIVE_KEYS`. The bootstrap runbook order is therefore `migrate -> provision (§2.7) -> start service`. The `AllowNoApiKeysForBootstrap` escape hatch is permitted ONLY outside Production (dev/test); it is FORBIDDEN (ignored/rejected) in Production. This check belongs with the pepper/readiness step (§2.4, after persistence).

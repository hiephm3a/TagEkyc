## 1. Intent
Establish the authoritative, immutable, versioned RAW-EXPORT POLICY CATALOG - the single source of truth a later enforcement slice (88B) reads to gate raw export. 88A stores policy versions (mode, purpose, retention, consent-requirement, controller + jurisdiction attributes), an allowed raw-class subset (a child table), and the SYSTEM-DERIVED typed approval requirements 88B's activation gate will check (derived via a VERSIONED rule set - closed selectors, migration-seeded, read by both C# and a DB trigger, and IMMUTABLY BOUND to each policy version so an old policy is never re-interpreted by a newer rule set) via an APPEND-ONLY store whose lifecycle reaches only Draft/CatalogApproved/Abandoned. **88A is deliberately INERT and DB-enforced: it builds no runtime capability, moving the design forward with zero risk of raw flowing.**

## 2. Scope - ALLOWED (inert catalog only)

### 2.1 Raw-Class taxonomy/registry [GREENFIELD shape] - ATOMIC, no catch-all
- A closed, provider-neutral enum of ATOMIC raw-class IDENTIFIERS. No aggregate/catch-all class (a broad class like "DG-groups"/"face-matching-material" could allow more than an approver intends - TIP-82R already learned `face_matching.chip_image` == DG2/chip raw, not a separate loophole). Proposed atomic set: `ChipDg1`, `ChipDg2Portrait`, `ChipDg13`, `ChipDg15`, `ChipSod`, `AaChallenge`, `AaResponse`, `LiveSelfieImage`, `LivenessMedia`, `HandSignatureImage`. Identifiers only (no bytes/format - agent-side), no vendor-specific classes.
- `HandSignatureImage` IS included as a known taxonomy identifier (TIP-82R explicitly lists hand-signature image as a possible raw class). The taxonomy enumerates POSSIBLE classes a policy may allow; whether a class is actually captured is agent-side/Slice-2 (an unknown/uncaptured class fails-closed there). No "if recon confirms" hedge in the enum.
- **Exclusions:** `face_match_score` / `liveness_result` are DERIVED scores, NOT raw BIO - do NOT include them as raw classes.

### 2.2 Raw Export Policy version aggregate [GREENFIELD EF, append-only]
- An insert-only `raw_export_policy_versions` row keyed `(PolicyId, PolicyVersion)`. Content = ATTRIBUTES only (the data that DERIVES requirements; NO artifact refs - those are fulfillment/88B):
  - `Mode` (TIP-82R: `external_export_only_no_retain` / `encrypted_export_packet` / `encrypted_raw_vault_retained`), `Purpose`.
  - retention = `RetentionProfileRef` (nullable) + `RetentionPurposeCode` (nullable) - NO inline duration/TTL (a duration is a retention PROFILE the ref points to; the `RetentionSchedule` REQUIREMENT is the approval ARTIFACT fulfilled in 88B - distinct).
  - `ConsentRequirement` (enum `Required`/`NotRequired`) - the single source for whether a `ConsentArtifact` requirement is derived.
  - **[R:H2] NO concrete `Recipient`** (the 88B GRANT decides which onboarded client may use a policy; a policy may CONSTRAIN via `RecipientCategory`/`RecipientAssuranceRequirement`).
  - controller + jurisdiction ATTRIBUTES: `ControllerRole`, `ControllerEntityRef`, `ControllerJurisdiction`, `RecipientJurisdiction`, `ProcessingInfrastructureJurisdiction`, `TransferScenarioCode`. **[R:Codex-M1] NO `CrossBorderAssessmentRef`, NO `TransferLegalBasisRef` on the version** - a cross-border-assessment REF is a fulfillment ARTIFACT (88B's fulfillment table, not an immutable-then-unfillable column); `TransferLegalBasis` as a policy VALUE is `TransferLegalBasisCode` (no `-Ref`), as an artifact it is fulfillment/88B. The version holds only the jurisdiction ATTRIBUTES that DERIVE the `CrossBorderAssessment` requirement (2.3).
  - **[R:GPT-H1] the BOUND derivation rule set: `RequirementRuleSetId` + `RequirementRuleSetVersion`** (immutable, NOT null, **non-null composite FK -> `raw_export_requirement_rule_sets`** so a version can never bind a nonexistent rule set) - the exact rule-set version whose selectors generated this policy's requirement rows. **[R:round6-H1 - stamped, not caller-chosen] the value is resolved+stamped by AddVersion/the DB to the CURRENT AUTHORING rule set (2.3), NOT taken from caller input** - so a new policy cannot bind an OLD/weaker historical rule set to dodge a newer requirement. NEVER re-read as "current" AFTER stamping. CatalogApprove's completeness trigger and 88B both re-derive against THIS bound version, so a later migration seeding RuleSet N+1 cannot change what an already-authored version means or whether it approves.
- **[R:M2] allowed raw-class subset = a CHILD TABLE** `raw_export_policy_allowed_classes (PolicyId, PolicyVersion, RawClass)` - NOT a JSON/array column (so the DB enforces per-element domain + uniqueness + non-empty): FK to the version, composite-unique `(PolicyId, PolicyVersion, RawClass)`, a `CHECK`/domain that `RawClass` is in the closed taxonomy (2.1).
- **[R:round6-P1 - Content is IMMUTABLE and this is DB-ENFORCED, not just claimed]** the version row + its allowed-class rows + its requirement rows are INSERT-ONLY, and a content change is a NEW `(PolicyId, PolicyVersion+1)` set. Moving from a JSON column to CHILD tables opened a hole: append-only triggers block `UPDATE`/`DELETE` but NOT a later `INSERT` of an extra child row (which would expand the raw-class subset / add a requirement to an already-authored - or already-`CatalogApproved` - version, in place, with no new version and outside the approval decision). **Fix: a `BEFORE INSERT` trigger on `raw_export_policy_allowed_classes` AND `raw_export_policy_requirements` requires the parent `raw_export_policy_versions` row to have been created in the SAME transaction** (compare the parent row's `xmin` to `pg_current_xact_id()`, **or an equivalent DB-enforced same-transaction guard** if a version needs different casting/handling - the INVARIANT to test is exact: no child insert outside the parent's creating transaction), so child rows are ONLY ever written atomically inside the `AddVersion` transaction - no post-creation and no post-approval append is possible by any path incl. raw SQL. NO `Status` column (status DERIVED, 2.3).

### 2.3 Lifecycle = Draft / CatalogApproved / Abandoned (append-only records, status DERIVED) [GREENFIELD]
- **[R:P2 semantic rename] "Approved" in 88A = CATALOG/DESIGN approval, NOT legal/production activation.** It means the catalog author approved the policy CONTENT+declarations. Runtime activation (`Active`) + the check that the required approval ARTIFACTS are actually provided = 88B. Use the term `CatalogApproved` so a DPO reader does not think a policy is legally live.
- **[R:round8-H1 - ONE closure record, PK-enforced XOR (not two race-prone tables)]** a version is closed (CatalogApproved OR Abandoned) via a SINGLE insert-only `raw_export_policy_closures` row: `PolicyId`, `PolicyVersion`, `ClosureType` (`CatalogApproved` | `Abandoned`, CHECK), `ClosedAtUtc`, `ClosedByPrincipalId`, `DecisionRef`. **PRIMARY KEY `(PolicyId, PolicyVersion)`** - so a version can have EXACTLY ONE closure, and the approve-vs-abandon XOR is a NATURAL DB constraint: two concurrent closures of the same version -> one wins on the PK, the other fails (write-skew impossible - no cross-table "does the other row exist?" check that two concurrent txns could both pass). **FK to the policy version** (no orphan closure). This REPLACES the v0.9 two-table `raw_export_policy_approvals`+`raw_export_policy_abandonments` design, whose per-table triggers were not concurrency-safe.
  - **Effective status DERIVED (single closure row):** closure `ClosureType=CatalogApproved` => `CatalogApproved`; closure `ClosureType=Abandoned` => `Abandoned`; NO closure row => `Draft`. Mutually exclusive by construction (PK). NO in-place status flip.
- **[R:round7-M1 + round8 - closure provenance hardened + honestly scoped]** `ClosedAtUtc` is **DB-generated (`transaction_timestamp()`), NOT accepted from the caller** (a raw-SQL insert cannot back-date it); `ClosedByPrincipalId` and `DecisionRef` are `NOT NULL` + **non-empty/non-whitespace CHECK** (for a `CatalogApproved` closure `DecisionRef` = the approval decision ref; for an `Abandoned` closure `DecisionRef` = a stable abandonment decision ref - a governance record, not free-text-optional). HOWEVER: 88A has no authenticated principal surface yet (that is 88B, which surfaces `PrincipalId` into `AuthenticatedClientContext`), so a runtime/raw-SQL insert can still ASSERT any `ClosedByPrincipalId` value. The brief states this honestly: **in 88A the closer identity is APPLICATION-ASSERTED, NOT tamper-proof actor evidence.** 88B binds the principal from the authenticated context (or routes closure through a controlled DB function) to make it authoritative; 88A does not overclaim it.
- **88A authoring lifecycle = Draft / CatalogApproved / Abandoned.** NO `Active`/`Retired` concept exists in 88A (no enum value, no type, no column, no table state) - those are runtime-activation states = 88B. `Abandoned` is NOT a runtime state; it is an authoring-terminal for a Draft that was discarded, and it builds no capability.
- **[R:round7-P1 - open-Draft-on-rule-change: the decision, made explicit] a stale open Draft must be closable, and the "must meet CURRENT rules" gate is 88B, not 88A.** Two forces collide: a version is bound to its authoring rule set (temporal stability, 2.2), yet OQ-4 allows only one open Draft and a new version needs the latest CLOSED (CatalogApproved OR Abandoned, 2.4) - so without a close path, a Draft bound to `RuleSet v1` would deadlock once `RuleSet v2` is seeded (can't approve safely, can't replace). Resolution (deliberate, not silent):
  - A `CatalogApproved` closure validates against the version's BOUND rule set (temporal, immutable) - and `CatalogApproved` is INERT design-approval, provably not live. **The "a policy must satisfy the CURRENT-authoring rule set to go LIVE" gate belongs to 88B activation** (88B refuses to activate a version whose bound rule set is not the current MAX - a stale policy must be re-authored to reach production), so no weakening reaches a live export even if a stale Draft is CatalogApproved in inert 88A.
  - **an `Abandoned` closure row** closes a Draft WITHOUT approving it (same `raw_export_policy_closures` table, `ClosureType=Abandoned`). Derived state: `Abandoned` (terminal). This gives the re-author path: abandon the stale `v1` Draft, then `AddVersion` a `v2`-rule-bound replacement.
  - the XOR is structural (ONE closure PK per version) - not a race-prone pair of cross-checking triggers. The chain rule (2.4) treats a version that HAS a closure row (either type) as "closed" so a new version may be added; `GetLatestCatalogApprovedVersion` = the latest version whose closure is `ClosureType=CatalogApproved` (Abandoned excluded).
- **[R:P1/M1 - DECLARATION separate from FULFILLMENT (the load-bearing fix)]** a policy version DECLARES its required approval types via an insert-only `raw_export_policy_requirements` row: `(PolicyId, PolicyVersion, RequirementType)` where `RequirementType` in `{ LegalApproval, ConsentArtifact, Dpia, CrossBorderAssessment, RetentionSchedule }`, with a **non-null composite FK `(PolicyId, PolicyVersion)` -> the policy version** (no orphan requirement - like allowed-classes and closures) and UNIQUE `(PolicyId, PolicyVersion, RequirementType)`. **This is DECLARATION only - immutable, no `ArtifactRef`.** The actual artifact FULFILLMENT (the ref proving each requirement was provided, which often arrives AFTER authoring) is an append-only fulfillment table + the presence-at-activation check - **both DEFERRED to 88B** (an append-only table cannot hold a "nullable-until-provided" ref; a later ref is a NEW appended fulfillment row, not an UPDATE). 88A stores WHAT is required; 88B records WHAT was provided + enforces it at activation.
- **[R:GPT-H1 - requirements derive on the RIGHT axes, not Mode alone; and are SYSTEM-DERIVED, not author-declared]** `Mode` describes storage/handling and only drives the STRUCTURAL requirements; the LEGAL requirements derive from policy ATTRIBUTES. The author supplies ATTRIBUTES; the SYSTEM computes the mandatory `RequirementType` declarations at `AddVersion` (the author cannot omit a mandatory one - removing the "author drops Dpia" risk). The derivation rule (encoded in the BOUND versioned rule-set tables, read by both the C# derivation and the DB trigger - see below):
  - **Global raw-export invariant:** `LegalApproval` is required for EVERY raw-export policy (stated as a catalog invariant, NOT pretended to be Mode-derived).
  - **Structural (Mode-driven):** `encrypted_export_packet` -> `RetentionSchedule`; `encrypted_raw_vault_retained` -> `RetentionSchedule` + `Dpia`; `external_export_only_no_retain` -> none structural.
  - **Contextual (attribute-driven):** `ConsentRequirement == Required` -> `ConsentArtifact` (`ConsentRequirement` is the single source; a lawful-authority/other-basis policy sets it `NotRequired` and NO `ConsentArtifact` is derived); **[R:Codex-P2 - cross-border rule PINNED] `HomeJurisdiction = VN`; if any of `ControllerJurisdiction`/`RecipientJurisdiction`/`ProcessingInfrastructureJurisdiction` is non-null AND != `VN` -> `CrossBorderAssessment` requirement derived. A null jurisdiction that the rule needs is INCOMPLETE (rejected at CatalogApprove).** Declared HERE because a version is immutable after CatalogApprove; 88B only checks the FULFILLMENT.
  - **[R:Codex-P2 + round6-P2 - EXACTLY the derived set: reject BOTH missing AND extra]** the requirement set must EQUAL the set the bound rule set derives - the completeness trigger rejects a version that is MISSING a derived-mandatory requirement AND one that carries an EXTRA declaration NOT in the derived set (e.g. `ConsentRequirement=NotRequired` + a stray `ConsentArtifact` row = a contradiction that would otherwise slip through). No "optional extras". Requirement declarations are unique per `(PolicyId, PolicyVersion, RequirementType)`. Per-rule tests: global `LegalApproval` always; mode -> retention/DPIA; consent-field -> `ConsentArtifact`; foreign-jurisdiction -> `CrossBorderAssessment`; and an extra-non-derived-requirement version -> rejected.
  - **[R:H1 enabler + GPT-H1 - a VERSIONED rule set, not a single mutable table] two seeded reference tables** so BOTH the C# derivation AND the DB completeness trigger (2.5) read ONE single source (no C#-vs-SQL drift) AND a policy version can pin the exact rules it was authored under:
    - `raw_export_requirement_rule_sets (RuleSetId, RuleSetVersion, MigrationRef, HomeJurisdictionCode)` - one row per rule-set version; PK `(RuleSetId, RuleSetVersion)`; strictly INSERT-ONLY (no lifecycle column). A rule change is a NEW `(RuleSetId, RuleSetVersion+1)`, never an edit. **[R:round7-H1 - a SINGLE system rule-set family, not a caller-chosen one] `RuleSetId` is a fixed system CONSTANT `RAW_EXPORT_REQUIREMENTS`** - NOT an `AddVersion` input, and a `CHECK` (or single-value domain) forbids any other `RuleSetId` from ever being seeded. This closes the cross-family downgrade (else a caller could bind a NEW policy to a weaker `LEGACY` family whose MAX happens to lack DPIA); "current-authoring resolved by the DB" is only well-defined with one family. `HomeJurisdictionCode` still lets policy VALUES vary; the family constant is about the requirement-RULE lineage. `HomeJurisdictionCode` (e.g. `VN`) lives IN the versioned rule-set metadata (NOT hard-coded in the evaluator) with a **canonical country-code format CHECK + non-empty**, so the `AnyJurisdictionForeign` selector means what the BOUND rule set says (a later rule set could set `SG`) and old policies stay stable. **[R:round6-H1] the CURRENT-AUTHORING rule set is DERIVED = `MAX(RuleSetVersion)`** of the constant family (seeding version N+1 automatically makes it current - no supersession column, no publications table, rule-set rows stay strictly insert-only). AddVersion binds ONLY that current-authoring (max) version; a NEW policy cannot bind an older version (a downgrade), but versions ALREADY bound to an older set stay valid + are still evaluated under it. **[R:round8-M1 - a rule-set version is published ATOMICALLY] the rule-set header row AND all its `raw_export_requirement_rules` rows MUST be seeded in ONE migration transaction (they become visible together, or the migration rolls back entirely)** - so `MAX(RuleSetVersion)` can never point at a header whose rules are not yet visible (which would derive a weak/empty requirement set). Defensively, AddVersion/the derivation binds a rule-set version ONLY if it has >= 1 rule row; a headerless-or-ruleless rule-set version can never be an authoring source (tested).

## 6. Open questions
- OQ-1b [RESOLVED = append-only]: lifecycle via an append-only CLOSURE record (`raw_export_policy_closures`, status derived from `ClosureType`), not an in-place status flip. Version rows insert-only.
- OQ-2 [RESOLVED = declaration-only, fulfillment deferred]: `raw_export_policy_requirements` is DECLARATION-ONLY `(PolicyId, PolicyVersion, RequirementType)` - NO `ArtifactRef`/`ArtifactVersion`. The artifact-fulfillment table (with the refs) + presence-enforcement are 88B (a later ref is a new appended fulfillment row, never an UPDATE of an append-only requirement row). Mandatory requirements are SYSTEM-DERIVED (2.3), not author-declared.
- OQ-3 [RESOLVED = Draft/CatalogApproved/Abandoned]: 88A domain + DB = Draft/CatalogApproved/Abandoned; `Active` (production activation) is 88B's migration + activation logic. `Abandoned` is an authoring-terminal, not a runtime state.
- OQ-4 [LOCKED = one open Draft, closable]: one open Draft per PolicyId; a new `AddVersion` is rejected until the latest is CLOSED (CatalogApproved OR Abandoned) - linear chain, no branching, and a stale Draft is closable via abandon (no deadlock). Tested.
- OQ-5 [RESOLVED]: retention on the policy = `RetentionProfileRef` + `RetentionPurposeCode` (nullable) - NO inline duration; the `RetentionSchedule` RequirementType is the distinct approval ARTIFACT proving the schedule was approved (88B checks it at activation: no-retain => no profile / NO_RETAIN; packet/vault => valid profile ref + RetentionSchedule fulfilled).
- OQ-6 [RESOLVED = versioned + bound]: the derivation rules are a VERSIONED rule set (`raw_export_requirement_rule_sets` + `raw_export_requirement_rules`, closed selector enum) IMMUTABLY BOUND to each policy version via `RequirementRuleSetId`/`RequirementRuleSetVersion`; a rule change = a new rule-set version (migration-only, append-only); C#/SQL evaluate the same selectors over the bound set (parity-tested). An already-authored policy never changes meaning when a newer rule set is seeded.
- OQ-7 [RESOLVED = DB-enforced chain]: the version-chain invariants (first=1, contiguous, latest-CLOSED = has a closure row of either type, one-open-Draft) are enforced by a `BEFORE INSERT` trigger + per-`PolicyId` advisory lock on `raw_export_policy_versions`, not repo-only - raw SQL cannot gap/fork the chain.
- OQ-8 [RESOLVED = current-authoring only, no downgrade]: a NEW policy version binds ONLY the current-authoring rule set (= `MAX(RuleSetVersion)` per `RuleSetId`, derived - no lifecycle column, rule-set rows stay strictly insert-only), resolved by the DB - not caller input; an older (non-max) rule set is bindable by nothing new but keeps validating the policies already bound to it. `HomeJurisdictionCode` is rule-set metadata (VN today), so a future jurisdiction change is a new rule-set version, not a silent evaluator change.
- OQ-9 [RESOLVED = content atomic-with-AddVersion]: child rows (`allowed_classes`/`requirements`) are insertable ONLY inside the version's creating transaction (`xmin` vs `pg_current_xact_id()`, or equivalent same-txn guard), so "content immutable" is DB-true - no post-creation/post-approval append can widen a version. The requirement set must EQUAL (not superset) the derived set (missing AND extra both rejected).
- OQ-10 [RESOLVED = single family + abandon]: `RuleSetId` is the fixed constant `RAW_EXPORT_REQUIREMENTS` (no cross-family downgrade); a stale open Draft is closable via an `Abandoned` closure (`raw_export_policy_closures`), resolving the one-open-Draft deadlock; a version is CatalogApproved XOR Abandoned XOR Draft - the XOR guaranteed by the closure PK (concurrency-safe), not by cross-table triggers.
- OQ-11 [RESOLVED = stance on stale-bound approval]: 88A `CatalogApprove` validates against the version's BOUND rule set (temporal stability) and is INERT design-approval; the "must satisfy the CURRENT rule set to go LIVE" gate is 88B activation (a stale-bound version must be re-authored to activate) - so no rule-weakening reaches a live export via a stale 88A approval.
- OQ-12 [RESOLVED = provenance honestly scoped]: the closure `ClosedAtUtc` is DB-stamped (`transaction_timestamp()`), `ClosedByPrincipalId`/`DecisionRef` non-empty; in 88A the actor identity is APPLICATION-ASSERTED (not tamper-proof) - authoritative principal binding is 88B (authenticated context / controlled DB function). The brief does not overclaim it.
- OQ-13 [RESOLVED = single closure table, PK-XOR]: CatalogApprove and Abandon are two `ClosureType`s of ONE `raw_export_policy_closures` row (PK `(PolicyId,PolicyVersion)`), so the XOR is a natural DB constraint - a concurrent approve+abandon race resolves on the PK (exactly one wins), with NO cross-table "does the opposite row exist?" read to write-skew. Replaces the v0.9 two-table design whose per-table triggers were not concurrency-safe. A real-concurrency test is required.
- OQ-14 [RESOLVED = atomic rule-set publication]: a rule-set header + its rules are seeded in ONE migration transaction (all-or-nothing visibility), and AddVersion/derivation refuses to bind a rule-set version with zero rule rows - so `MAX(RuleSetVersion)` can never select a partially-published (weak/empty) rule set.

## 1. Intent
Produce a FROZEN, consumer-agnostic golden neutral-proof vector (the exact `{jws, publicKeyJwk, expectedTrustAnchor{kid,fingerprint}, challengeInput.expectedChallenge, fingerprint, kid, 19 claims, identityRef inputs, constants}` a real TagEkyc completion emits, plus the independent binding inputs a conformant consumer must check against), a self-verify regression test that keeps it valid, and a conformance doc pinning the byte-exact rules. Capture-once-and-freeze: ES256 is non-deterministic (random k) + `signedAt=UtcNow` + per-run GUIDs, so the vector is a STATIC snapshot the consumer only VERIFIES, never regenerates. NOT a signing-behavior change; test/fixture/doc only.

## 2. Scope - ALLOWED

### 2.1 Generate + commit the frozen fixture(s)
- Adapt `Tip67BNeutralProofVerificationTests.cs` `CompleteChallengeBoundSessionAsync` (:186-250) with PINNED inputs (fixed subjectRef, challenge, requiredChecks) to run the real pipeline once and emit the `EvidencePackageVerificationViewDto` + JWS.
- Capture VERBATIM into a committed JSON fixture: `jws` (compact `header.payload.signature`), `publicKeyJwk` (the 4-field `{"crv":"P-256","kty":"EC","x":..,"y":..}` byte-exact - fingerprint depends on it), `publicKeyFingerprint` (`sha256:<hex>`), `kid`, ALL 19 decoded payload claims (proofVersion..resultHashCanonicalizationScheme incl. `identityRef`/`assuranceLevel`/`challenge`/`sessionId`/`packageId`/`resultHash`/`signedManifestHash`/`signedAt`), and `identityRefInputs` (`clientApplicationId` GUID:N + `subjectRef` + label `tip-67b-identity-ref-v1`) so the consumer can independently recompute identityRef.
- **[review P1-01] `challengeInput.expectedChallenge` as an INDEPENDENT input:** freeze the ORIGINAL challenge the caller passed at create-session, SEPARATE from the signed `challenge` claim. A conformant consumer verifies the proof's `challenge` == its OWN expected challenge (transaction/session binding), NOT the claim compared to itself. (Reading `challenge` from the decoded JWS and comparing it to itself proves nothing - a proof for a different transaction would still "pass".)
- **[review P1-02] `expectedTrustAnchor: { kid, publicKeyFingerprint }` as an INDEPENDENT input:** the pinned trust anchor a consumer holds OUT-OF-BAND. A conformant consumer MUST pin `(kid, fingerprint)` and only then accept the embedded `publicKeyJwk` - it MUST NOT blindly trust whatever key the JWS embeds (classic embedded-key forgery: an attacker signs a fake proof with their own key + embeds their own JWK -> self-verifies). The captured JWK is frozen TEST material; the ANCHOR is the expected pinned kid+fingerprint.
- **Vectors:** (a) **Medium-assurance MANDATORY** (single `CaptureQuality` check) - the primary vector exercising full verify + identityRef + resultHash; (b) **High-assurance RECOMMENDED-IF-FEASIBLE** (NFC+FaceMatch+Liveness -> `High`, CalculateAssuranceLevel :586-603) to cover the multi-check `requiredChecks`/`completedChecks` array shape + the High value. If the multi-check evidence-append path is straightforward in the test infra, include it; else defer High to a follow-up (do NOT block the Medium vector on it).
- **[self-check] Synthetic inputs only - NO real PII:** subjectRef/challenge/clientReference are fixed SYNTHETIC test values (e.g. `subject-ref`, `opaque challenge: 123`); the fixture must contain NO real patient/BHXH/identity data.
- **[review P1-05] Provenance metadata** on each vector (for future refreeze reviews): `generatedByCommit` (SHA), `generatedAt`, `vectorName`, `assuranceTarget`, `signerKid`, `publicKeyFingerprint`, `sourceTestMethod` (the test that generated it), and a `syntheticNoPii: true` note.
- **Location = consumer-agnostic contract artifact** (e.g. `docs/contracts/golden_neutral_proof_vector*.json` or a `contracts/` fixtures dir) - NOT buried in a test project; the consumer copies it as a FROZEN LITERAL (no runtime/test-time dependency on any TagEkyc assembly).

### 2.2 Self-verify regression test (locks the fixture valid + drift-alarm)
- A TagEkyc test loads the committed frozen fixture and runs a verify path based on `Tip67BReferenceVerifier` (:460-621): asserts (i) JWS signature valid against the FROZEN `publicKeyJwk` (ASCII signing bytes, P1363 raw R||S 64-byte, unpadded b64url); (ii) recomputed fingerprint == frozen fingerprint; (iii) `resultHash` recomputes from the 16-field preimage (label `tip-67b-neutral-proof-result`, JCS); (iv) `identityRef` recomputes from `identityRefInputs` (label `tip-67b-identity-ref-v1`); (v) all 19 claims present + mirror the DTO.
- **[review P2] EXTEND the verifier, do NOT reuse as-is:** the current `Tip67BReferenceVerifier` only MIRRORS `identityRef` from the DTO/JWS - it does NOT recompute it from `clientApplicationId + subjectRef + label`. This slice MUST ADD the identityRef-recompute assertion (iv). Extension is test-only (no production surface).
- **[review P1-01] Challenge binding (vi):** the self-verify passes `challengeInput.expectedChallenge` EXTERNALLY into the verifier (as `Tip67BReferenceVerifier.Verify` already takes `expectedChallenge`) and asserts the proof's `challenge` claim == that external value - NOT read from the decoded claim and compared to itself.
- **[review P1-02] Trust-anchor pin (vii):** the self-verify pins the embedded key against `expectedTrustAnchor` - asserts `jws.kid == expectedTrustAnchor.kid` AND recomputed-fingerprint == `expectedTrustAnchor.publicKeyFingerprint` BEFORE trusting the embedded JWK; does NOT trust the embedded key blindly.
- **[review P1] Cheap NEGATIVE self-tests (mandatory):** (n1) same fixture + WRONG `expectedChallenge` => verify FALSE; (n2) same fixture + WRONG `expectedTrustAnchor` (kid or fingerprint) => verify FALSE. These prove the binding checks actually gate, not just the happy path.
- **[review P2] Generator is a MANUAL one-shot, NOT run in `dotnet test`:** if a generator test/tool exists, it is excluded from the normal test run (e.g. `[Fact(Skip=...)]` or a separate console tool). The normal self-verify test ONLY loads the frozen committed JSON and verifies; a `dotnet test` run MUST NOT write/modify `docs/contracts/golden_neutral_proof_vectors.json`.
- Purpose: the fixture is only useful if guaranteed-valid; this test locks it AND, if signing/canonicalization ever drifts, the test breaks -> a loud signal the wire contract changed (and the vector + all consumers must be re-frozen).
- **[self-check] Drift-alarm mechanism (precise):** the signature-verify checks the FROZEN `header.payload` bytes (stays green even if production canon changes, since the bytes are frozen); the DRIFT alarm fires via the RECOMPUTE paths - `resultHash` (re-canonicalizes the frozen 16-field preimage via production `EvidenceCanonicalization`) and `identityRef` (re-hashes from `identityRefInputs`). If production JCS/label/formula drifts, the recompute != the frozen value -> test fails -> re-freeze signal. So the self-verify MUST include the recompute assertions, not only signature-verify + field-presence.

### 2.3 Conformance doc (byte-exact rules the consumer must match)
- A short doc pinning the byte-exact rules. It MUST include:
  - **The exact 19 payload claims** (verbatim from `BuildProofClaimJson` - the captured fixture is authoritative; BUILD confirms names+order from source): `proofVersion`, `purpose`, `sessionId`, `identityRef`, `packageId`, `packageVersion`, `canonicalizationScheme`, `hashAlgorithm`, `result`, `assuranceLevel`, `requiredChecks`, `completedChecks`, `evidenceEngines`, `signedAt`, `challenge`, `signedManifestHash`, `resultHash`, `resultHashAlgorithm`, `resultHashCanonicalizationScheme`.
  - **The exact 16-field resultHash preimage** (= the 19 minus the 3 `resultHash*` fields): `proofVersion`, `purpose`, `sessionId`, `identityRef`, `packageId`, `packageVersion`, `canonicalizationScheme`, `hashAlgorithm`, `result`, `assuranceLevel`, `requiredChecks`, `completedChecks`, `evidenceEngines`, `signedAt`, `challenge`, `signedManifestHash`; `resultHash = "sha256:" + lowerhex(SHA256(UTF8(label + "\n" + JCS(preimage))))`, label = `tip-67b-neutral-proof-result`.
  - **identityRef byte-exact:** `identityRef = "sha256:" + lowerhex(SHA256(UTF8("tip-67b-identity-ref-v1" + "\n" + clientApplicationId:N + "\n" + subjectRef)))` - pin the `sha256:` prefix, the `\n` separators, the `:N` (32-hex no-dash) GUID form, UTF-8.
  - JCS canonicalization (`EvidenceCanonicalization` - Ordinal key sort, RFC-8785 number formatting, control-char escaping); unpadded base64url; signing input hashed over ASCII bytes; signature raw P1363 R||S 64-byte (NOT DER); protected header `{"alg":"ES256","kid":..}` (JCS-ordered, NO `typ`).
  - **`signedAt` format:** UTC, rendered `yyyy-MM-dd'T'HH:mm:ss.fffffff'Z'` = **7 fractional digits, value truncated to MICROSECONDS before formatting** (so the 7th digit is typically 0). Do NOT describe it as "µs-precision 7-digit" (unit-ambiguous).
  - **JWK fingerprint over CANONICAL form:** the 4-field JWK JCS order (`crv`<`kty`<`x`<`y`); `fingerprint = "sha256:" + lowerhex(SHA256(UTF8(canonical-4-field-JWK)))`. The consumer need NOT preserve the fixture file's whitespace/key-order - it MUST canonicalize the 4-field JWK identically before fingerprinting, and MUST NOT hash the expanded 10-field JWKS metadata object.
- **[self-check] identityRef = FORMULA, not value:** the vector's `clientApplicationId` is the TEST business-client GUID. The golden vector proves the consumer's identityRef-recompute ALGORITHM is correct against known inputs->output; in PRODUCTION the consumer uses its OWN registered `clientApplicationId`. The doc must state: freeze/test the formula (`sha256(label\n{clientAppId:N}\n{subjectRef})`), not the specific test GUID.
- **[review P1] SEMANTIC binding rules (not just crypto validity)** the doc MUST teach:
  - **Challenge binding:** the consumer MUST compare the proof's `challenge` claim against its OWN expected challenge (the value it passed at create-session / its binding_nonce), NOT read `challenge` from the proof and accept it. A valid signature over the WRONG challenge is a wrong-transaction proof.
  - **Trust anchor:** the consumer MUST pin `(kid, publicKeyFingerprint)` OUT-OF-BAND (from the JWKS endpoint / a pinned anchor) and reject a proof whose embedded key does not match; it MUST NOT trust whatever `publicKeyJwk` the JWS embeds. Blind-trusting the embedded key is a forgery vector.

## 3. STOP / NOT
- CAPTURE-ONCE-AND-FREEZE - NO regenerate-and-diff (ES256 random-k + `UtcNow` signedAt + per-run GUIDs make it non-reproducible); the consumer VERIFIES the frozen vector, never regenerates.
- NO consumer-specific concept - this is a GENERIC neutral-proof vector (Tip67B neutral proof); no "SignFlow"/"consent" in the fixture or doc.
- NO production signing/canonicalization change - test/fixture/doc ONLY; if the self-verify test forces a signing change, STOP + report (that is a wire-contract change, out of scope).
- NO fixed-key requirement - the captured `publicKeyJwk` is frozen TEST material (no committed P12/PEM needed); the vector's trust ANCHOR is the INDEPENDENT `expectedTrustAnchor` (pinned kid+fingerprint), NOT blind trust of the embedded JWK (2.1-P1-02). Production consumers MUST pin kid+fingerprint out-of-band, never trust whatever key the JWS embeds.
- NO runtime/test-time dependency on a TagEkyc assembly for the CONSUMER - the vector is copied as frozen literals (the self-verify test here MAY use internals; the consumer must not).
- NO migration / no schema / no endpoint change.

## 5. Review tier & attacks
Tier-2 (test/fixture, but it is the cross-impl CONFORMANCE ANCHOR). Attacks: (a) is the fixture actually captured from the REAL signer (not a hand-written guess)? (b) is the frozen JWK byte-exact so the fingerprint verifies? (c) does the self-verify test truly exercise signature-verify + resultHash + identityRef recompute (not just field presence)? (d) any production signing change sneaked in? (e) any consumer-specific leakage? (f) does the doc capture EVERY byte-exact subtlety (ASCII bytes, P1363, unpadded b64url, signedAt truncated-to-microseconds rendered with 7 fractional digits, JCS order) a consumer could diverge on? (g) does the vector prove EXTERNAL challenge binding (not self-referential) and out-of-band trust-anchor pinning (not blind-trust of the embedded JWK)?

### 2.4 Fulfillment (POLICY-scoped only; two-event append-only state machine)
- **Scope:** 88B1 fulfillment covers ONLY the POLICY-scoped requirements `{LegalApproval, Dpia, CrossBorderAssessment, RetentionSchedule}` (S0). `ConsentArtifact` is subject-scoped and is NOT a policy-level fulfillment here.
- Append-only `raw_export_fulfillments` keyed `(PolicyId, PolicyVersion, RequirementType)`. **Exactly two event types:**
  - `Accepted` - creates/replaces the effective artifact; carries `ArtifactRef`, `ArtifactVersion`, `ValidFromUtc`, `ValidUntilUtc?`, optional `SupersedesRevision` (the current effective revision it replaces; null = first acceptance). Requires an authorized recorder (S2.3/S2.7).
  - `Withdrawn` - invalidates the current effective revision; carries `TargetRevision`; NO artifact/validity fields.
  - Schema: `FulfillmentEventId`, `(PolicyId,PolicyVersion,RequirementType)`, `Revision`, `EventType`, `SupersedesRevision?`(Accepted), `TargetRevision?`(Withdrawn), artifact+validity (Accepted only, all required there), `RecordedByPrincipalId`(M2), `FulfillmentDecisionRef`(non-empty), `RecordedAtUtc`(DB-stamped).
- **State-machine rules (DB-enforced):** `ValidUntilUtc > ValidFromUtc` when present; artifact fields iff `Accepted`.
- **[R:Codex-P3 - RENEWAL path, no self-lock after expiry/withdraw] a `TargetRevision` (Withdrawn) MUST be the current EFFECTIVE revision** (stale-target reject). **But a new `Accepted` may RENEW: its `SupersedesRevision` targets the LATEST `Accepted` revision of the aggregate EVEN IF that revision is expired or withdrawn** (so an expired/withdrawn fulfillment can always be renewed by appending a NEW accepted artifact); `SupersedesRevision=null` ONLY for the very first acceptance (no prior `Accepted` exists). What is FORBIDDEN is "resurrection" = making an OLD withdrawn/superseded revision effective again by reference (reusing its artifact); appending a NEW `Accepted` artifact is renewal, allowed. Tests: expired -> new `Accepted` renews (succeeds); withdrawn -> new `Accepted` renews (succeeds); reusing/pointing an old withdrawn revision as effective -> rejected.
- **[R:GPT-4 - deterministic resolution] current effective fulfillment:** walk the aggregate by `MAX(Revision)`: if the latest event is `Accepted` AND currently valid (within `[ValidFromUtc,ValidUntilUtc)`) => that is the effective artifact; if the latest event is `Withdrawn`, or the latest `Accepted` is expired => there is NO effective fulfillment (NEVER silently fall back to an older `Accepted`; renewal requires a NEW `Accepted`). Deterministic by `MAX(Revision)`.

(If runtime root-management is ever intended, it must be added to S2.7 with self-escalation / last-root / concurrency tests - explicitly OUT of 88B1.)

- **[R:Codex-P4 - rule-set publish lock coordination]** the resolver's "no `Authorized` after a rule-set publish" holds ONLY if a rule-set publish and the resolver read are serialized. 88A rule sets are MIGRATION-seeded (migration-only, current = `MAX(RuleSetVersion)`), so an app advisory lock does NOT block a migration by itself. Pin ONE protocol (R:round9-P1): **a NAMED advisory lock (fixed key, e.g. `hashtext('raw_export_rule_set_publication')`) with MODE distinction** - the authorization resolver takes it in SHARED mode; a rule-set-publish migration takes it in EXCLUSIVE mode (`pg_advisory_xact_lock` for exclusive vs `pg_advisory_xact_lock_shared` for shared). Shared readers do not block each other but an exclusive publish DOES block/serialize against readers, so no `Authorized` is emitted concurrent with a publish. Rule-set publication remains a deployment/maintenance-window operation (documented). A migration/procedure test asserts the publish path takes the exclusive lock and the resolver takes the shared lock. (SERIALIZABLE is NOT used here - the advisory-lock-with-mode protocol is the single chosen approach.) **[R:verify-A2 - accuracy] TODAY there is NO runtime rule-set publisher** (rule-set versions are 88A migration-only/offline), so the resolver's SHARED lock currently has no live EXCLUSIVE counterpart and `StaleRuleSet` is actually enforced by the bound-vs-`MAX(RuleSetVersion)` comparison in the resolver read. The shared lock is a FORWARD-LOOKING hook: IF a runtime publisher is ever added it MUST take the exclusive lock on `RuleSetPublishLockKey`, else the shared lock protects nothing. The brief does not claim a live runtime publish-race is closed by the lock today - version-comparison closes it; the lock is future-proofing.

## 6. Open questions (remaining)

Not a blocker for 88B1 (Policy+Global is sufficient for the hospital single-client trial).

## 0. Grounding (on-code, CC-verified @ 2c684cf)
- **[Subject consent is GREENFIELD]** no existing per-subject/per-session consent ARTIFACT in TagEkyc; 88A's `ConsentArtifact`/`ConsentRequirement` is a POLICY requirement attribute, not a queryable subject record; TIP-82's consent-chain is agent-gate behaviour (`7f50e73`), not a persisted per-session artifact the authorization path can read.
- **Session (recon):** `VerificationSession` = `Id` (Guid, EkycSessionId), `ClientApplicationId` (owner), `SubjectRef` (string, the subject), `State` (`Completed` = exportable). Consent binds the full `ConsentScopeHash` (2.5).
- **88A `ConsentArtifact` = per-subject export consent required**; 88B1 EXCLUDED it from policy-level fulfillment; THIS slice owns its per-subject satisfaction.
- **Reuse 88B1's proven append-only mechanisms:** monotonic contiguous `Revision` + per-aggregate `pg_advisory_xact_lock` + NULL-safe `UNIQUE(key,Revision)` + `SECURITY DEFINER` append functions (owner = deployment role, runtime `EXECUTE` only) + actor from `SET LOCAL tagekyc.actor_principal_id` (fail-closed) + append-only UPDATE/DELETE-deny trigger. Same 3-role deployment gate posture.

## 1. Intent
Establish an append-only SUBJECT-EXPORT-CONSENT record so TIP-88B3 can, per attempt, resolve a CURRENTLY-VALID subject consent for the exact `ConsentScopeHash` and read its current `ConsentedRawClasses`. This slice RECORDS + RESOLVES consent state; it does NOT authorize export, choose purpose/recipient, derive requested classes, evaluate class coverage, assign authorization denial causes, issue a permit, or read raw bytes. **Purpose + recipient are SERVER-DERIVED; class-coverage is TIP-88B3's.**

## 2. Scope - ALLOWED
### 2.1 Subject-consent record (greenfield append-only)
- **[aggregate identity = full semantic scope WITHOUT raw classes]** deterministic **`ConsentScopeHash`** (2.5) over EXACTLY six typed fields: `VerificationSessionId` + authoritative `SubjectRef` (server-loaded) + `PolicyId` + `PolicyVersion` + `PurposeCode` (server-derived, 2.1a) + `RecipientClientApplicationId` (server-derived, 2.1b). Raw classes are NOT in scope identity nor lock key - they are immutable CHILD CONTENT of each `Granted` revision. Two consent events `{Granted, Withdrawn}` with monotonic contiguous `Revision` PER SCOPE.
- **[2.1a PurposeCode - server-derived closed taxonomy]** `PurposeCode` is a CLOSED enum, **server-derived, NEVER request-body input.** **Slice-1 pins the taxonomy to a SINGLE value: `SubjectRawBiometricExport`.** The caller does not choose it. Any future multi-purpose taxonomy is a SEPARATE review/TIP (not this slice).
- **[2.1b RecipientClientApplicationId - server-derived]** a Guid recipient identity, **NOT caller-selectable.** The consent command LOADS it from the locked `VerificationSession` row: `RecipientClientApplicationId = VerificationSession.ClientApplicationId` (same-owner). If a deployment temporarily retains it as an assertion parameter, it is EXACT-MATCH validated against the locked session owner and FAILS CLOSED on mismatch. An external/TSP recipient different from the session owner is DEFERRED to a separate delegation/governance TIP.
- **[refs, disambiguated]** a `Granted` row: `SubjectConsentRecordId` (PK/handle), `ConsentScopeHash`, `VerificationSessionId`, `SubjectRef`, `PolicyId`+`PolicyVersion`, `PurposeCode`, `RecipientClientApplicationId`, `ConsentTextVersion` + `ConsentTextContentHash` (2.6), `ExternalConsentArtifactRef` (2.6), `DecisionRef` (2.6), `ValidFromUtc`, `ValidUntilUtc?`, `CapturedAtUtc`, `CapturedByPrincipalId`, `TargetRevision` = NULL (2.4). `ConsentedRawClasses` = immutable child content of this revision.
- **[`ConsentedRawClasses`]** NON-EMPTY, unique per class, CLOSED raw-class taxonomy, written in the SAME transaction as the `Granted` parent (xmin same-txn guard), NO later append (a change = a NEW `Granted` revision). **This slice STORES + RETURNS the consented set; it does NOT compare it against any required set** (coverage is TIP-88B3's, 2.3).
- **[A1 - current state = latest event OVERALL, NO fallback]** the effective consent for a `ConsentScopeHash` is derived from the LATEST event overall for that scope:
  - latest = `Withdrawn` => NON-EFFECTIVE (no fallback);
  - latest = `Granted` but currently EXPIRED or NOT-YET-VALID (`now ∉ [ValidFromUtc, ValidUntilUtc)`) => NON-EFFECTIVE (no fallback to any earlier revision);
  - latest = `Granted` and currently valid => EFFECTIVE;
  - **only a LATER new `Granted` revision reactivates/replaces consent** (never an older revision resurfacing).
  - **Test plan (mandatory):** rev1 `Granted` valid `[t0, t_far)`; rev2 `Granted` valid `[t1, t2)` with `t2` near; after `t2`, rev2 is the latest and is expired => the scope is NON-EFFECTIVE; **rev1 must NOT reappear.** Plus: latest `Withdrawn` after a valid `Granted` => non-effective; a subsequent `Granted` => effective again.
- **[validity]** `ValidFromUtc == CapturedAtUtc == transaction_timestamp()` (DB time; NO caller back-/future-dating); `ValidUntilUtc?` nullable, `ValidUntil > ValidFrom` CHECK; half-open `[ValidFromUtc, ValidUntilUtc)`.

### 2.2 Consent-recording / withdraw authority (SECURITY BOUNDARY, locked)
- **[DEDICATED consent-authority aggregate; do NOT reuse/mutate 88B1's `raw_export_control_authorities`.]** Greenfield append-only.
  - **Key: `(AuthorityPrincipalId, ClientApplicationId, AuthorityType)`.**
  - **`AuthorityType` in `{SubjectConsentRecorder, SubjectConsentWithdrawer}`** (recorder governs consent `Granted`; withdrawer governs consent `Withdrawn`).
  - **`EventType` in `{Granted, Revoked}`** (authority-lifecycle; distinct from CONSENT events).
  - **Fields:** `Revision` (monotonic contiguous), `ValidFromUtc`, `ValidUntilUtc?`, `GrantedByPrincipalId` (deployment write-boundary actor), `RecordedAtUtc` (DB-stamped), `TargetRevision` (2.4), **`DecisionRef`** (governance/approval reference for the grant or revoke, 2.4).
  - **Lifecycle [latest-event-overall, NO FALLBACK - same rule as consent 2.1 A1]:** the effective authority for a key is derived from the LATEST event overall for that key:
    - latest = `Revoked` => INACTIVE (no fallback);
    - latest = `Granted` but currently EXPIRED or NOT-YET-VALID (`now ∉ [ValidFromUtc, ValidUntilUtc)`) => INACTIVE (**no fallback to any earlier authority revision**);
    - latest = `Granted` and currently valid => ACTIVE;
    - **only a LATER new `Granted` reactivates** (never an older revision resurfacing). A `Revoked` targets the current effective `Granted` revision (2.4). DB `transaction_timestamp()` stamps.
    - **Test plan (mandatory):** authority rev1 `Granted` valid `[t0, t_far)` (long); rev2 `Granted` valid `[t1, t2)` (short); after `t2`, rev2 is the latest and expired => the authority is INACTIVE and **rev1 must NOT reappear** => a runtime consent write by that principal is DENIED. Plus: latest `Revoked` => inactive; a later `Granted` => active again.
- **[authority separation - fail-closed]** an export grant confers NO consent authority; SESSION OWNERSHIP confers NO consent authority; runtime CANNOT self-grant.
- **[deployment-only authority management]** authority `Granted`/`Revoked` is DEPLOYMENT-ONLY: the runtime role has NO EXECUTE on the authority-management function(s). Runtime may only READ the authority aggregate + append CONSENT events.
- **[runtime consent write gated on current authority]** a runtime consent `Granted`/`Withdrawn` is allowed ONLY if a CURRENT effective `SubjectConsentRecorder`/`SubjectConsentWithdrawer` authority exists for the acting principal (checked under the authority-aggregate lock, 2.4); else DENIED. `CapturedByPrincipalId`/withdrawer come from the authenticated write-boundary context (M2 `SET LOCAL`), NEVER the request body.

### 2.3 Resolver (consumed by TIP-88B3) [ambient-txn, no caller SubjectRef, NO class evaluation]
- **`ResolveSubjectExportConsentForAuthorization(verificationSessionId, policyId, policyVersion)`** - THREE parameters only. The resolver DERIVES the remaining scope fields itself: `PurposeCode` = the Slice-1 constant `SubjectRawBiometricExport` (2.1a), and `RecipientClientApplicationId` = the owner read from the LOCKED `VerificationSession` row (2.1b). NO `purposeCode`/`recipientClientApplicationId` parameters, NO `requiredRawClasses`, NO caller `SubjectRef` - so a caller cannot influence ANY scope field.
  - **[FINAL - pinned resolver LOCK CONTRACT]** REQUIRES an ambient transaction, then in this exact order: (1) ACQUIRE (or REUSE, if the caller already holds it) the **`VerificationSession` ROW LOCK**; (2) read the authoritative `SubjectRef` + owner from that locked row; (3) derive `RecipientClientApplicationId` (= locked owner) and `PurposeCode` (constant); (4) compute the `ConsentScopeHash` (2.5); (5) ACQUIRE the **consent-scope SHARED advisory lock**. **BOTH the session row lock AND the consent-scope lock remain HELD until the CALLER's commit/rollback** (the resolver never releases them). This preserves the global order `session row -> consent scope` (2.4) and is why the resolver can never be called before the session is locked.
  - Resolves the CURRENT effective consent (2.1 A1) under those locks.
  - RETURNS `{ Effective: bool, SubjectConsentRecordId, ConsentScopeHash, Revision, ConsentedRawClasses[], PurposeCode, RecipientClientApplicationId, SubjectRef, ValidFromUtc, ValidUntilUtc, EvaluatedAtUtc, Provenance{ConsentTextVersion, ConsentTextContentHash, ExternalConsentArtifactRef, DecisionRef}, Cause }`.
  - **Does NOT accept/evaluate `requiredRawClasses`; does NOT compute coverage; does NOT emit authorization denial causes.** `Cause` is a FACTUAL consent-state cause (`missing` / `withdrawn` / `expired`) - NOT an authorization decision (TIP-88B3 maps it, 88B3 B5).
- **REQUIRES an ambient EF/DB transaction; NEVER self-opens/self-commits** (throws `RAW_EXPORT_SUBJECT_CONSENT_REQUIRES_AMBIENT_TRANSACTION`).

### 2.4 Write lock order + TargetRevision
- **TargetRevision semantics:** consent `Withdrawn` and authority `Revoked` carry a `TargetRevision`; consent/authority `Granted` carry `TargetRevision = NULL`. A `Withdrawn`/`Revoked` `TargetRevision` MUST equal the CURRENT effective `Granted` revision, checked UNDER the aggregate lock; a STALE target (not the current effective revision) is REJECTED with a stable conflict (`RAW_EXPORT_CONSENT_STALE_TARGET_REVISION` / `..._AUTHORITY_STALE_TARGET_REVISION`).
- **[B2-4 - EXACT event row-shape]**
  - **Consent `Withdrawn` row:** the six typed scope columns + `ConsentScopeHash`; `Revision` = next contiguous; **`TargetRevision` NOT NULL** = the current effective `Granted` revision; `WithdrawnByPrincipalId` = the acting principal from the authenticated WRITE-BOUNDARY context (`SET LOCAL tagekyc.actor_principal_id`), **NEVER the request body**; `RecordedAtUtc` = `transaction_timestamp()` (DB time); `DecisionRef` = the governance/approval reference authorizing the withdrawal (typed, non-empty when a governance step exists, else NULL); optional `ExternalConsentArtifactRef` withdrawal-evidence ref (**bytes OUT**). **NO `ConsentedRawClasses` children** (class children belong to `Granted` only); no `ValidFromUtc`/`ValidUntilUtc` (a `Withdrawn` is an instantaneous transition, not a validity-bearing event).
  - **Authority `Revoked` row:** the key columns `(AuthorityPrincipalId, ClientApplicationId, AuthorityType)`; `EventType = 'Revoked'`; `Revision` = next contiguous; **`TargetRevision` NOT NULL** = the current effective authority `Granted` revision; the acting principal from the DEPLOYMENT write-boundary context (never a request body); `RecordedAtUtc` = `transaction_timestamp()`; `DecisionRef` = the governance/approval reference authorizing the revocation; `ValidFromUtc`/`ValidUntilUtc` NULL (not a validity-bearing event).
  - **[B2-2] Authority `Granted` row:** the key columns `(AuthorityPrincipalId, ClientApplicationId, AuthorityType)`; `EventType = 'Granted'`; `Revision` = next contiguous; **`TargetRevision = NULL`**; `GrantedByPrincipalId` = the acting principal bound to the DEPLOYMENT write-boundary context (never a request body, never runtime-supplied); `RecordedAtUtc` = `transaction_timestamp()` (DB time); `ValidFromUtc` (DB time) + `ValidUntilUtc?`; **`DecisionRef`** = the governance/approval reference authorizing granting this authority (typed, non-empty when a governance step exists, else NULL).
  - **Consent `Granted` row:** `TargetRevision = NULL` (full shape in 2.1).
  - **[B2-3] DB CHECKs, split PER AGGREGATE (the two event vocabularies are distinct):**
    - **consent-event table** - `EventType IN ('Granted','Withdrawn')`; `EventType = 'Granted' => TargetRevision IS NULL`; `EventType = 'Withdrawn' => TargetRevision IS NOT NULL`.
    - **consent-authority table** - `EventType IN ('Granted','Revoked')`; `EventType = 'Granted' => TargetRevision IS NULL`; `EventType = 'Revoked' => TargetRevision IS NOT NULL`.
    - Neither table admits the other's event vocabulary (no `Withdrawn` on the authority aggregate, no `Revoked` on the consent aggregate).
- **[B2-1 - the two consent commands have DIFFERENT session-state gates]**
  - **Consent `Granted` command:** (1) lock the authoritative `VerificationSession` row; (2) reload owner, `SubjectRef`, `State`; (3) **require `State == Completed`** (consent to export is captured over a COMPLETED verification); (4) derive `PurposeCode` (Slice-1 constant) + `RecipientClientApplicationId` (= locked owner); (5) require a CURRENT effective **`SubjectConsentRecorder`** authority under the authority-aggregate lock; (6) acquire the consent semantic-scope lock (2.5); (7) append `Granted` + its immutable class children (same txn, xmin guard); (8) commit.
  - **Consent `Withdrawn` command:** (1) lock the authoritative `VerificationSession` row; (2) reload owner, `SubjectRef`, current `State`; (3) **require ONLY that the session EXISTS - do NOT require `State == Completed`.** *(A subject must be able to withdraw consent after the session leaves `Completed`; gating withdrawal on `Completed` would trap consent in a non-revocable state.)* (4) derive the SAME `PurposeCode` + `RecipientClientApplicationId` (= locked owner) so the scope resolves to the SAME `ConsentScopeHash`; (5) require a CURRENT effective **`SubjectConsentWithdrawer`** authority under the authority-aggregate lock; (6) acquire the SAME consent semantic-scope lock (2.5); (7) validate `TargetRevision` == the current effective `Granted` revision (else stale conflict, 2.4); (8) append `Withdrawn` (no class children); (9) commit.
  - **Mandatory tests:** `Granted` recorded while `Completed` -> session transitions to `Expired` / `Cancelled` / `TechnicalTerminal` -> **`Withdrawn` SUCCEEDS** in each of those three states; the resolver then reports `withdrawn` / NON-EFFECTIVE. A stale `TargetRevision` STILL conflicts in the non-`Completed` case (the state relaxation does not weaken revision checking). Because purpose/recipient/`SubjectRef` are re-derived from the LOCKED session row, a wrong-subject / wrong-owner / wrong-scope withdrawal is structurally impossible. Also: `Granted` attempted in any non-`Completed` state -> REJECTED (all 6 non-Completed states asserted).
- **Global lock order: `session row -> consent-authority aggregate -> consent semantic scope -> append event -> commit`. NEVER acquire the session lock AFTER a consent-scope lock.** The resolver takes the SAME canonical consent-scope lock identity. Required two-transaction tests: record vs authority-revoke; record/withdraw vs session-transition; resolver vs withdrawal; stale-`TargetRevision` withdraw vs concurrent renew.

### 2.5 ConsentScopeHash + advisory-lock mapping (server/DB-computed) + golden vectors
- **`ConsentScopeHash` COMPUTED server/DB-side, NEVER request-supplied.**
- **Algorithm:** `ConsentScopeHash = SHA-256( NS || LP(f1) || ... || LP(f6) )`, `NS = "tagekyc:consent-scope:v1"` (UTF-8), `LP(x)` = 4-byte big-endian length prefix + canonical bytes. Field order + canonical encoding: (1) `VerificationSessionId` = 16-byte RFC-4122 Guid (big-endian); (2) authoritative `SubjectRef` = UTF-8 NFC bytes; (3) `PolicyId` = 16-byte Guid; (4) `PolicyVersion` = 8-byte big-endian integer; (5) `PurposeCode` = closed-enum canonical token, UTF-8; (6) `RecipientClientApplicationId` = 16-byte Guid.
- **[A4 - ONE pinned advisory-lock mapping] do NOT use `hashtext(...)`.** The advisory-lock key is EXACTLY: **the FIRST 8 BYTES of the server-computed `ConsentScopeHash`, interpreted as a SIGNED BIG-ENDIAN int64**, passed to the PostgreSQL **single-`bigint`** form `pg_advisory_xact_lock(<int8>)`. There is NO two-int alternative and NO other mapping. Fixed, documented, and identical for `Granted`, `Withdrawn`, and the resolver.
- **Typed columns remain AUTHORITATIVE:** every match/uniqueness predicate compares the six typed columns, not only the hash - a collision/spoof CANNOT merge unrelated scopes.
- **[A4 - independent golden vectors] the test suite MUST pin byte-exact golden vectors, EACH independently, for:** (i) RFC-4122 Guid -> 16-byte encoding; (ii) `SubjectRef` -> NFC UTF-8 bytes; (iii) `PolicyVersion` -> big-endian encoding; (iv) `PurposeCode` -> canonical token bytes; (v) `RecipientClientApplicationId` Guid -> 16-byte encoding; (vi) the six-field concatenation -> SHA-256 output; (vii) the SHA-256 output -> the **exact signed big-endian int64 advisory-lock key** (first 8 bytes). Each vector pins byte-exact values frozen at build/test time, and asserts **C#/SQL parity** (the C#-computed int64 must equal the SQL-computed int64 for the same scope).

### 2.6 Consent provenance & trusted-recorder contract
- **`ExternalConsentArtifactRef`** = MANDATORY opaque external-evidence ref ASSERTED by the trusted recorder. **Bytes-OUT** (no raw artifact bytes stored). If the server cannot independently resolve+verify it, it is a **trusted-recorder ASSERTION, not server verification** (stated explicitly on the record).
- **`ConsentTextVersion` + `ConsentTextContentHash`** = MANDATORY. **BUT: absent a server-side IMMUTABLE consent-text registry, `ConsentTextContentHash` is classified a TRUSTED-RECORDER ASSERTION, NOT server proof of the text shown.** The brief does NOT claim server proof of shown text. A future OPTIONAL registry MAY verify `ConsentTextVersion -> expected hash` (then it becomes server-verified); until then it is recorder-asserted.
- **`DecisionRef`** = the governance/approval decision reference that authorized CAPTURING this consent; typed non-empty when a governance step exists; OMITTED (nullable) if none.

### 2.7 DB privilege / SECURITY DEFINER function security (locked)
- **Runtime role has NO direct `INSERT`/`UPDATE`/`DELETE`/`TRUNCATE`** on the consent-authority, consent-event, or consent-class tables. Runtime ONLY `EXECUTE`s the intended consent-append function(s).
- **`PUBLIC` EXECUTE revoked** on all consent functions; EXECUTE granted only to the runtime capability role (append fns) and deployment role (authority-mgmt fns).
- **SECURITY DEFINER function manifest:** expected OWNER = a NON-LOGIN deployment role; FIXED `search_path` (e.g. `pg_catalog`); NO unsafe dynamic SQL. Readiness FAILS on ACL / owner / `search_path` / `prosecdef` / function-signature DRIFT.
- **Direct-SQL negative tests:** connecting AS the runtime role, prove it CANNOT INSERT/UPDATE/DELETE/TRUNCATE any consent table, CANNOT EXECUTE authority-mgmt fns, CANNOT self/other-grant authority. **No post-parent child insertion** (children only in the parent's transaction; a later child INSERT is rejected by the xmin same-txn guard).
- **[FINAL - SPLIT event invariants; there is NO generic DB-enforced `session-Completed` invariant]** a privileged/direct INSERT path (even if privilege leaked) is enforced PER EVENT TYPE, never by one blanket trigger:
  - **Consent `Granted` path enforces:** the session EXISTS; authoritative owner + `SubjectRef` LOADED from the locked session row; **`State == Completed`**; a CURRENT effective `SubjectConsentRecorder` authority; the EXACT derived `PurposeCode` + `RecipientClientApplicationId` (matching the locked owner); contiguous `Revision`; the full `Granted` row shape + provenance (`ConsentTextVersion`/`ConsentTextContentHash`/`ExternalConsentArtifactRef`/`DecisionRef`, `TargetRevision = NULL`); immutable same-transaction class children.
  - **Consent `Withdrawn` path enforces:** the session EXISTS; authoritative owner + `SubjectRef` LOADED from the locked session row; **NO `Completed` requirement**; a CURRENT effective `SubjectConsentWithdrawer` authority; the EXACT derived `PurposeCode` + `RecipientClientApplicationId` and the resulting `ConsentScopeHash`; contiguous `Revision`; **`TargetRevision` == the current effective `Granted` revision**; the `Withdrawn` row shape; **NO class children**.
  - **Explicitly:** a withdrawal after the session reaches `Expired` / `Cancelled` / `TechnicalTerminal` **MUST SUCCEED** through the intended append function and **MUST NOT be blocked by a generic `Completed` trigger**. Any blanket "session must be Completed" trigger over the consent-event table is a DEFECT - the `Completed` predicate belongs to the `Granted` path ONLY.

## 6. Review tier & attacks
Tier-1. Attacks: (a) caller injects PurposeCode/Recipient (must be server-derived)? (b) an export consumer self-records consent or self-grants a recorder authority (authority bypass)?

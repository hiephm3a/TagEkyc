## 0. Why this study + what grounds it

TIP-71 proved the ViewFaceCore engine RUNS correctly (pts5) on **web photos**. It did NOT prove SeetaFace6 is **discriminative enough for production 1:1** on the REAL operating condition — a **low-res, compressed CCCD chip DG2 photo vs a live selfie** — nor that the server's **0.80 threshold** is the right operating point for SeetaFace's score scale (native threshold ~0.62). A single same-demographic impostor pair scored 0.505; the FALSE-ACCEPT TAIL on many hard pairs is unmeasured. Betting production (and building TIP-72 liveness on top) before measuring this is out of order. This study measures it + decides the engine + the threshold.

## 1. Intent

Measure the **genuine vs impostor score distributions** for face verification (live selfie vs CCCD chip DG2) for **ViewFaceCore/SeetaFace6 (pts5)** AND **dlib (DlibDotNet)** on the SAME pairs, compute **ROC / EER / FAR@FRR**, and decide: **(a) is SeetaFace adequate for production 1:1 on real DG2, or switch to dlib? (b) what is the calibrated operating-point threshold?** Output feeds a TIP-70 server-threshold follow-on. **The shared study deliverable contains AGGREGATE STATISTICS ONLY; local working artifacts may include gitignored per-pair score CSVs under the §3 privacy policy. No raw faces leave the local machine.**

## 2. Scope — ALLOWED

1. **Calibration harness (eval tool in the agent repo `TagEkyc.CaptureAgent`)** — reuses the `IFaceMatchScoreSource` seam. Inputs a LOCAL, gitignored folder `{ opaque-identity-id → dg2.<img> + selfie(s).<img> }`. Runs BOTH engines (ViewFaceCore pts5 + a new dlib `IFaceMatchScoreSource`) on every **genuine** pair (same id: DG2 vs its selfie(s)) and **impostor** pair (DG2_i vs selfie_j, i≠j), with a **same-demographic impostor subset flagged** (the hard negatives).
2. **Metrics computed:** per engine — genuine & impostor score distributions (mean/std/percentiles + histogram bins), **ROC/DET, EER, FAR@{FRR=1%,5%}**, and a **suggested operating threshold**. A side-by-side SeetaFace-vs-dlib comparison.
3. **dlib wiring** — add `DlibFaceMatchScoreSource : IFaceMatchScoreSource` (DlibDotNet) so the same harness runs both engines apples-to-apples (also de-risks the dlib fallback).
4. **Two output tiers (round-2 — separate them):** (a) **local working artifacts** = a gitignored per-pair score CSV keyed by salted-per-run opaque ids (no images/names/paths) — local-only, never committed, never sent to the Contractor; (b) **shared deliverable** = aggregate-ONLY markdown (distributions/histograms, ROC, EER, FAR/FRR tables, threshold). The harness MUST NOT persist images or embeddings beyond the in-run buffer (no-raw, reuse `SensitiveByteBuffer`).
5. **Score-scale invariant (round-1):** every engine MUST report `{rawScore, scoreDirection (higher-is-similar | lower-is-similar), normalized [0,1] similarity, monotonic-mapping-rule, which-scale-the-threshold-is-on}`. ROC computed AFTER declaring higher-is-better. dlib's distance→similarity conversion documented + bounded/validated (NOT an arbitrary `1-distance`). **Engine thresholds are ENGINE-SPECIFIC** — the normalized [0,1] is for common plotting/reporting only; engines are compared by ROC/AUC/EER + FAR-constrained FRR, **NEVER** by equal normalized threshold value (`SeetaFace 0.80 == dlib 0.80` is wrong).
6. **Failure / quality buckets (round-1, NOT optional):** report SEPARATELY from the ROC — no-face, multi-face, DG2-decode-fail, live-decode-fail, poor-quality/tiny-face/blur/pose/occlusion, timeout/exception (FTA/FTM). Report `comparable-pair rate`, `failure-rate-by-reason`, `FRR-among-comparable` AND `FRR-including-non-comparable-failures`. A good EER on successful comparisons while (e.g.) 20% no-face ≠ production-adequate. Failures must NOT be hidden as score-0 or silently dropped.
7. **Threshold-selection protocol (round-1, anti-optimism):** **identity-disjoint** split (split by IDENTITY, not by pair → no leakage). Threshold chosen on a dev/calibration split; the acceptance metric reported on a held-out test split (or k-fold by identity). NEVER use the same pairs to both pick the threshold and claim the performance. Small N → label the whole study **exploratory / calibration-signal only**.
8. **dlib = EVAL-ONLY (round-1 boundary):** the dlib `IFaceMatchScoreSource` lives in an **eval-only project/console inside the agent repo** — the PRODUCTION agent binary MUST NOT silently gain a second native engine dependency. Production stays ViewFaceCore-only unless the study decides a switch.
9. **Same-demographic hard-negative bins (round-1/2):** define NON-identifying bins (age-band / sex / presentation) only if consented + recorded LOCALLY; report subgroup FAR/FRR where N permits, **marked exploratory unless N is sufficient**. **Minimum-cell suppression:** no subgroup/demographic table is shared below a minimum cell count — small groups are merged or suppressed. **Balance is enforced by the pair SAMPLER (not just docs)** so one identity does not generate many correlated impostor pairs that dominate the metric.
10. **Phased datasets (§5).**

## 3. STOP / privacy ceiling — defect if violated

- **NO raw faces / embeddings committed or logged** — input dir gitignored; output = aggregate scores keyed by opaque id only (no names, no images, no base64). Reuse `SensitiveByteBuffer` zeroing; no embedding persistence.
- **CONSENT required for real biometric (Phase 1)** — only consented volunteers' CCCD-DG2 + selfie; no scraping of real people; document consent basis. Public-benchmark images (Phase 0) stay local, not committed.
- **NO production-readiness claim from a small N** — report N, and the confidence honestly (small N = directional, not certified). No "production-grade" verdict without an adequate sample.
- **NO server change in this study** — the threshold value lives server-side (TIP-70); this study only RECOMMENDS a value → a separate TIP-70 follow-on changes it. No manifest/proof/semantic change.
- **NO score fudging** — faithful normalization per engine scale (SeetaFace identity, dlib its own scale → both to [0,1] similarity); do not rescale to hit a target.
- **NO liveness / presentation-attack-detection (PAD) / anti-spoofing — explicitly OUT OF SCOPE (TIP-72).** This study measures **1:1 IDENTITY DISCRIMINATION ONLY** (genuine person vs a DIFFERENT person). Its FAR/FRR/ROC say NOTHING about spoof-resistance: a printed photo / screen replay / mask of the GENUINE person would score as a high genuine match here yet is a presentation attack — that threat model is liveness/PAD (TIP-72, a separate engine + study), NOT this study. **A passing FaceMatch here does NOT imply the presented face was live.**
- **CSV privacy policy (round-1):** the per-pair scores CSV is **local-only, `.gitignored`, NEVER committed, NEVER sent to the Contractor**. It is biometric-DERIVED measurement + relationship data — opaque ids = study-local random/salted per run (NOT stable across studies); NO file paths / filenames / granular demographics / stable volunteer ids / embeddings / images. Only **aggregate histograms / ROC / EER / FAR-FRR tables** are shared; demographics only at aggregate group level (no per-person row).
- **Recommendation-packet boundary (round-1/3):** the study output is a RECOMMENDATION packet only — **no server threshold change, no production-grade claim, no model/vendor certification claim**; it CREATES a TIP-70 threshold-calibration follow-on if needed. **If it recommends switching to dlib, that is a RECOMMENDATION only — production dlib requires a SEPARATE productionization TIP (license/provenance packet, native packaging, no-raw controls, dependency guard, server-threshold follow-on). This study does NOT ship or authorize dlib in production.**
- **Phase-0 benchmark is NOT privacy-clean / NOT production evidence** — LFW (if used) = real public-figure faces, research-use, local-only, broad engine-sanity ONLY; never called privacy-clean or production proof.
- **HARD STOP before any Phase-1 real-biometric run (round-2 P1):** no real CCCD-DG2 / selfie collection until ALL are defined: consent text, local storage path, retention window, **deletion procedure (delete-after-eval)**, and who may access the local folder. Phase-1 is BLOCKED until this precondition is signed off.
- **`.gitignore` rule (round-2):** the agent/eval repo gitignores the calibration output folder + score files (e.g. `calibration_outputs/`, `*.scores.csv`, `*.pairs.csv`) so per-pair artifacts cannot be committed by accident.

## 4. Definition of Done / acceptance

- [ ] Harness runs both engines on the same genuine + impostor pairs (incl. the same-demographic hard subset); outputs the aggregate report (ROC/EER/FAR-FRR per engine + side-by-side) with NO raw faces/embeddings in output or logs.
- [ ] **Phase 0A (synthetic smoke) — REQUIRED:** harness, score-direction, engine calls, normalization, failure handling validated (NOT adequacy; NOT ROC unless the synthetic set has labeled same-identity pairs).
- [ ] **Phase 0B (public benchmark / LFW) — OPTIONAL:** if chosen, broad SeetaFace-vs-dlib sanity only (local-only, research-use, NOT privacy-clean, NOT production evidence). Phase-0 may REJECT/FLAG an engine; it CANNOT support a production-grade adequacy claim.
- [ ] **Phase 1 (real DG2-vs-selfie, consented) — REQUIRED for any production-grade adequacy recommendation** (gated by the §3 consent/retention HARD-STOP) → real-condition ROC + the false-accept TAIL on same-demographic impostors.
- [ ] **Hard-negative FAR reported + gated SEPARATELY:** overall-impostor FAR vs same-demographic/hard-negative FAR (+ per-identity/clustered analysis where N permits) — the acceptance target MUST NOT be satisfied by a large easy-impostor pool alone.
- [ ] **Correlated-pair robustness:** beyond the binomial-CI caveat, report a second view (identity-cluster bootstrap / per-subject false-accept exposure / max-or-percentile impostor false-accept by identity) so few-identities-many-pairs cannot make the CI overconfident.
- [ ] **Operating point carries its capture-quality envelope:** any recommended threshold names the quality-gate policy it assumes (min face size, blur/pose/crop, no-face/multi-face handling) — a threshold without its envelope is not portable.
- [ ] **Failure/quality buckets** reported separately from the ROC (FTA/FTM by reason; comparable-pair rate; FRR-among-comparable AND FRR-including-failures) — no hiding failures in score-0 or by dropping them.
- [ ] **Identity-disjoint dev/holdout split** (or k-fold by identity): threshold chosen on dev, acceptance metric reported on holdout; small-N labelled exploratory/calibration-signal-only.
- [ ] **Score-scale invariant** met per engine (raw + direction + normalized + monotonic-rule + threshold-scale); engine thresholds engine-specific; dlib distance→similarity documented/bounded.
- [ ] Per-pair CSV is local-only / gitignored / never-committed / never-sent; shared output = aggregate only.
- [ ] A **decision recorded**: SeetaFace adequate (+ recommended threshold) OR switch to dlib (+ threshold), with the FAR/FRR justification + **CI** + the N/confidence caveat.
- [ ] The recommended threshold + engine feed a registered **TIP-70 server-threshold follow-on** debt item (recommendation packet only — no server change in this study).

**Proposed acceptance target (Homeowner/security to confirm — an integrity/security operating TARGET, NOT a legal mandate):** a **low FAR** (proposed start **FAR ≤ 0.1%** on the impostor set incl. same-demographic) at an acceptable FRR (genuine rejects → ReviewRequired, human-handled). **FAR ≤ 0.1% is a TARGET, not a certified claim** — to CLAIM it, the **upper 95% binomial CI must also be ≤ 0.1%**; otherwise the result is directional / a threshold-candidate only. The report MUST include: N-genuine, N-impostor, observed FAR/FRR + **binomial CI**, **FRR @ FAR {0.1%, 0.5%, 1%}** (sensitivity), the ReviewRequired-rate, and the recommended operating point. Note impostor-pair **non-independence** (one identity → many correlated pairs) when reporting CIs.

## 5. Phased execution

**Phase 0A — synthetic smoke (PII-free-ish, first):** run both engines on synthetic faces (SFHQ MIT, local) → validates the harness, score-direction, engine calls, normalization, and failure handling. **NOT an adequacy claim, and NOT ROC/EER** unless the synthetic set has identity labels + genuine same-identity pairs (SFHQ-style data usually lacks same-identity variations) → use it for decode/detect/score-direction/failure-handling only.
**Phase 0B — optional public benchmark (engineering sanity):** OPTIONALLY run both engines on a standard protocol (e.g. LFW 6000-pair; images local, NOT committed, research-use terms) → broad SeetaFace-vs-dlib general-discrimination signal. **NOT privacy-clean, NOT production evidence** — engine sanity only; if SeetaFace is already far below dlib here, decisive early. NOT on the required path.

**Phase 1 — real DG2-vs-selfie (the real gate, consented):** Homeowner reads real CCCD chip DG2 (HN212) + captures a live selfie for each consented volunteer → the harness runs both engines on the real low-res-chip-vs-selfie pairs (genuine + same-demographic impostors) → the REAL-condition ROC + false-accept tail. Small N gives a directional signal; grow N for confidence. This is the operating condition web photos cannot represent.

**Execution constraint (honest):** the Contractor (remote) designs the harness + analyzes the reported stats; the **Homeowner runs the native harness locally** (Windows + native ViewFaceCore/dlib + the local images) and reports the AGGREGATE stats — exactly like the HN212 probe + the TIP-71 opt-in test. No raw biometric crosses to the Contractor.

## 6. Deliverable + decision

A **decision brief**: (a) engine choice (SeetaFace vs dlib) with the ROC/EER/FAR-FRR evidence; (b) the calibrated operating-point threshold; (c) N + confidence caveat (+ "binomial CI approximate / optimistic if impostor pairs correlated"); (d) any quality-gate needs (min face size / pose / crop) surfaced by the data; (e) a **reproducibility packet** — exact engine + model versions, **model-artifact hashes**, landmark profile (`pts5/Light`), detector config, preprocessing/crop/resize rules, and the dlib model/version/hash — so the calibrated threshold is reproducible later. → feeds the TIP-70 threshold follow-on + the production go/no-go for 1:1 FaceMatch.
